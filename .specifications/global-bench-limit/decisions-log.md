# Decisions log — global-bench-limit

- **2026-05-22** — Where should `maxBenchesGlobal` live on `UserPreferences`? → Nested under a new `benches` object: `UserPreferences.benches?: { maxGlobal?: number }`. Rationale: anticipates more bench-related app settings without re-organising later. (captured during /product-prd)
- **2026-05-22** — How should `0` and negative numbers be treated? → Invalid. Only `null`/absent means unlimited; the cap, when set, must be a positive integer >= 1. Rationale: avoids two ways to express "unlimited" and rejects nonsensical inputs at the validation boundary. (captured during /product-prd)
- **2026-05-22** — Do error-state / preparing benches count toward the cap? → Yes; any bench in the in-memory `benches` Map counts regardless of status. Rationale: matches the "workspace exists" mental model; user must clear errored benches to free the slot. (captured during /product-prd)
- **2026-05-22** — Render a global meter on the dashboard? → Yes, when a cap is set. Rationale: mirrors the per-project `ProjectTile` meter; gives the user advance signal before hitting the limit. (captured during /product-prd)
- **2026-05-22** — Performance budget for the cap check? → No measurable budget; check is O(1) on `benches.size`. (captured during /product-prd)
- **2026-05-22** — Accessibility target? → WCAG 2.1 AA via React Aria `Button` with `isDisabled` + associated description; meter with `aria-label`. Rationale: matches existing project convention. (captured during /product-prd)
- **2026-05-22** — Security expectations? → None new. Reuses existing `/api/settings` route (no auth, desktop single-user). Rationale: feature does not change the trust boundary. (captured during /product-prd)
- **2026-05-22** — Alignment findings AF-001 through AF-007 (all `level_gating_violation`, medium severity, covering FR-001, FR-002, FR-004, FR-007, FR-008, FR-012, NFR-002) → Dismissed in bulk. Rationale: each of these requirements has L1 coverage and is sufficiently narrow that intermediate L2 / L3 cases would not add meaningful signal beyond what the L1 unit tests + the existing L2 end-to-end journeys (TC-021, TC-022, TC-023, TC-024, TC-025) already exercise. The gating rule is enforced for execution order (L1 must pass before L2); it does not require every requirement to be authored at every level. If the team later finds a real gap, they can add the missing case as a follow-up; the dismissal is not permanent. (captured during /product-align)
