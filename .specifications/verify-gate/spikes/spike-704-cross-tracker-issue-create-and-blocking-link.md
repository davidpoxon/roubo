# Spike 704: How does Roubo create an issue and register an "is blocked by" relationship across GitHub, GHE, and Jira through the integration-plugin abstraction?

**Status:** Resolved · **Issue:** #704 · **Class:** decision · **Resolves:** prd.md FR-010/FR-011/NFR-005 (cross-tracker create + blocking-link); architecture.md:221-224 Open question (the cross-tracker spike that gates Phase 4); feasibility.md risk 1 (no cross-tracker issue-create + blocking-link capability) · **Implements:** FR-010, FR-011, NFR-005 · **Verified by:** (spike; no test cases) · **Gates:** #705 (tracker-action gateway), #706 (failed-case fix-issue filer) · **Recommendation:** adopt

## Objective and method

Phase 4 of the verify gate (failed-case fix-issue filing) needs three privileged tracker operations Roubo does not have today: create an issue, register an "is blocked by" relationship from the gate issue to the new fix issue, and close the gate issue on pass. The architecture isolates these in a new `TrackerActionGateway` (architecture.md:66) routed through `pluginManager.invoke`, capability-gated and audit-logged. Before that gateway is built (#705) this spike settles the unknowns the feasibility report flagged as the one high-risk item: whether each tracker can actually perform the create and the blocking-link write, and how a tracker that cannot should degrade.

This is a **decision** spike: the deliverable is this findings doc, not code. Building the `TrackerActionGateway`, the `FixIssueFiler`, the manifest capability-flag schema, the consent surface, or any plugin method is out of scope here and lands in #705 and #706. No change to `plugin-sdk/src/types.ts`, the manifest schema, or any plugin source is made by this spike.

Method: read the actual plugin and host code as the source of truth, and derive each tracker's create / blocking-link write surface from the request mechanism the plugin already uses for reads and for its existing writes (assign, transition). A capability is asserted "available" only where the plugin already calls the carrying mechanism (so the write is a near-neighbour of code that ships today) or where the tracker's documented API provides it; everything else is called out as a gap. The grounding files read:

- `plugin-sdk/src/types.ts` (`PluginContract`: every method optional) and `plugin-sdk/src/define-plugin.ts` (the `CONTRACT_METHODS` allowlist). There is no `createIssue` / `addBlockedBy` method and no `supportsCreateIssue` / `supportsBlockingLinks` flag today.
- `plugins/github-com/src/`: `index.ts` (method wiring), `methods/assign-issue.ts` and `methods/apply-transition.ts` (the existing privileged-write precedent), `github-request.ts` (the `octokit.graphql()` mutation mechanism), `github-fetchers.ts:178-216` and `normalize.ts:144-160` (the existing `blockedBy` READ graph walk).
- `plugins/ghe/src/index.ts` and its `methods/` directory (the current op surface).
- `plugins/jira-self-hosted/src/`: `index.ts`, `plugin.ts` (contract wiring; `jiraSearch` showing the `POST` write mechanism), `jira-client.ts` (`jiraFetch` supports `POST`/`PUT`), `link-types.ts` (the existing blocking link-type read + configurable names), `assignment.ts` / `transitions.ts` (existing write precedent).
- `server/services/plugin-manager.ts` (`invoke`, lines 1458-1515) and `server/services/audit-log.ts` (the `AuditLog` store), plus `schema/roubo-plugin.schema.json:53-57` (the manifest `capabilities` object) and `shared/plugin-consent-schema.ts` (the declared-permission consent model).
- `.specifications/verify-gate/{prd.md,feasibility.md,architecture.md}` for FR-010/FR-011/NFR-005 and the open questions, and `.specifications/verify-gate/issues.json` for the follow-on issue numbers.

## Findings per acceptance criterion

### AC1: per-tracker create-issue and "is blocked by" write APIs, and the gaps

The two privileged ops the gateway needs are `createIssue` and `addBlockedBy` (architecture.md:130-132). Roubo's server has no issue-create path anywhere today (feasibility.md:27); the only existing plugin writes are assign and transition. So both ops are net-new on every tracker. What differs per tracker is whether the write mechanism is already wired and whether the tracker's API supports the blocking-link write at all. GitHub is evaluated first, per the issue and the GitHub-first phasing (feasibility.md:74).

**GitHub (github-com).** Both ops are reachable through the `octokit.graphql()` mechanism `github-request.ts` already exposes (`githubRequest({ kind: "graphql", query, variables })`, line 224-226), which is the same path the blocking-link READ already uses (`github-fetchers.ts:242` issues a GraphQL query). The two existing privileged writes (`assign-issue.ts`, `apply-transition.ts`) go through the REST arm of the same helper (`PATCH /repos/{owner}/{repo}/issues/{issue_number}`), so a new GraphQL mutation slots in beside proven write code.

- **createIssue.** GitHub's GraphQL `createIssue` mutation (input `repositoryId`, `title`, `body`, `labelIds`, returns `issue { number url id }`) is GA and stable. The plugin already resolves a repository node id in its Projects v2 path, and `parseRepo` (`github-request.ts:90`) turns `owner/repo` into the pieces a `repository(owner:,name:){ id }` lookup needs. The REST `POST /repos/{owner}/{repo}/issues` is an equally valid alternative and avoids the node-id round-trip; either is a thin addition. No gap.
- **addBlockedBy (the blocking-link write).** This is the load-bearing finding. GitHub's issue-dependencies feature exposes the relationship Roubo already READS: `github-fetchers.ts:183-204` selects `issue.blockedBy(first: 10)` and `issue.blocking(first: 100)` connections, and `normalize.ts:144` walks them into `NormalizedIssue.blockedBy` / `blocks`. The READ is exercised and shipping. The corresponding WRITE is the `addSubIssue`-adjacent issue-dependency mutation pair (`addBlockedByIssue` / the `addBlockedBy`-family mutation on the issue-dependencies API): given the gate issue and the fix issue, it records "gate is blocked by fix". This mutation is the still-maturing surface: GitHub's issue-dependencies API has been moving from preview toward GA, so the gateway MUST treat its availability as runtime-discoverable, not assumed. The honest position for #705: probe it on the connected instance (a capability probe, mirroring the `host.capability.query` version-gate pattern the component SDK already defines in `plugin-sdk/src/types.ts:742-746`) and, if absent, degrade rather than assume. The READ shipping today is evidence the connection model works; it is not evidence the WRITE is enabled on a given instance. **Gap:** the write mutation's GA status is instance- and date-dependent; the gateway must verify, not presume.

**GitHub Enterprise (GHE).** The GHE plugin is **read-only today**: `plugins/ghe/src/index.ts` wires no `assignIssue`, no `unassignIssue`, and no `applyTransition` (compare github-com's `index.ts:33-36`, which wires all three). So GHE has no privileged-write precedent at all; both `createIssue` and `addBlockedBy` are greenfield there. Mechanically GHE is a near-identical port of github-com (same `github-request.ts`, same `octokit.graphql()` arm), so the same mutations would slot in the same way once written. The decisive caveat is **version-gating**: GHE ships the GraphQL schema of whatever GHE release the instance runs, and the issue-dependencies mutation is gated by that release. An older GHE simply does not expose the mutation in its schema. So GHE create is feasible whenever GHE is wired for writes, but GHE blocking-link is conditional on the instance's GHE version exposing the issue-dependencies mutation. **Gap:** GHE is write-unwired today, and its blocking-link write is version-gated by the GHE release; both must be discovered per instance, never assumed from github-com parity.

**Jira (jira-self-hosted).** Jira has the richest existing precedent. `jira-client.ts`'s `jiraFetch` already supports `POST` and `PUT` (`JiraFetchOptions.method`, line 4-8), `plugin.ts` already issues `POST /rest/api/2/search` for reads, and the plugin already performs privileged writes (`assignment.ts`, `transitions.ts`). And `link-types.ts` already READS the blocking relationship, with the link-type names configurable per project (`blocksLinkTypeName` / `isBlockedByLinkTypeName`, line 36-37) to tolerate renamed link types (TC-072).

- **createIssue.** `POST /rest/api/2/issue` with `{ fields: { project: { key }, issuetype: { id|name }, summary, description, labels } }`. The plugin already reads project keys, issue types (`listIssueTypes`, `plugin.ts:469`), and labels, so it has every field the create body needs. No gap, beyond mapping the gateway's `{ title, body, labels }` onto Jira's `summary` / `description` / `labels` field shape.
- **addBlockedBy (the blocking-link write).** `POST /rest/api/2/issueLink` with `{ type: { name: <blocks-family link type> }, inwardIssue: { key }, outwardIssue: { key } }`. Jira models blocking as one link type carrying two directional descriptions ("blocks" outward, "is blocked by" inward); `link-types.ts:38-44` already identifies that family from the configured names. The write reuses the exact configured `blocksLinkTypeName`, and the direction (gate is blocked by fix) decides which key is inward vs outward, the mirror of the read mapping at `link-types.ts:46-52`. No gap on a standard Jira; the one caveat is that the named link type must exist on the instance (the user can rename it, and `link-types.ts` already carries the configurable names to handle that), so the write should validate the link type resolves and degrade legibly if it does not.

**Summary (AC1).**

| Tracker              | createIssue                                                                                          | addBlockedBy (blocking-link write)                                                                                                                        | Write precedent today                               |
| -------------------- | ---------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------- |
| **github-com**       | GraphQL `createIssue` (GA) or REST `POST /issues`; node-id resolvable from existing Projects v2 path | issue-dependencies mutation; READ already shipped (`blockedBy`/`blocking` connections), WRITE unexercised; **GA status runtime-discoverable, must probe** | assign + transition via REST arm of `githubRequest` |
| **ghe**              | Same GraphQL/REST surface as github-com once writes are wired                                        | Same mutation as github-com, **version-gated by the GHE release**; absent on older GHE                                                                    | **none (read-only plugin today)**                   |
| **jira-self-hosted** | REST `POST /rest/api/2/issue` (fields already known to the plugin)                                   | REST `POST /rest/api/2/issueLink` with the configured blocks-family link type; READ already shipped                                                       | assign + transition via `jiraFetch` POST/PUT        |

The gaps that the gateway (#705) must carry: GitHub's blocking-link write GA status is runtime-discoverable; GHE is write-unwired and its blocking-link write is version-gated; Jira's blocking-link write depends on the named link type resolving on the instance. None of these is a blocker for GitHub-first delivery, which is exactly why the phasing is GitHub-first (feasibility.md:74).

### AC2: the capability flags and the degraded-mode behaviour

**Today's capability model is implicit.** Every method on `PluginContract` is optional (`plugin-sdk/src/types.ts:417`), and `definePlugin` only registers handlers for the methods a plugin actually provides (`define-plugin.ts:63-69`). A host call to an unimplemented method gets JSON-RPC `MethodNotFound`, which `pluginManager.invoke` translates from numeric `-32601` to the string `"MethodNotFound"` (`plugin-manager.ts:1506-1507`) so callers can detect it. So "does this plugin support op X" is answered implicitly: try the call, catch `MethodNotFound`. There is no explicit boolean flag for any contract method today, and the manifest's `capabilities` object is declared but **empty** (`schema/roubo-plugin.schema.json:53-57`: `"properties": {}`, `additionalProperties: false`), so no plugin declares any capability.

**Recommendation: add explicit boolean capability flags, do not rely on the implicit `MethodNotFound` model for these ops.** NFR-005 requires "a per-capability flag so an unsupported tracker degrades with a clear message" and "an absent capability yields a legible degrade, never a silent no-op" (prd.md:169-174). The implicit model is too weak for this for three reasons:

1. **It cannot distinguish "not implemented" from "implemented but unavailable on this instance."** GitHub's blocking-link write GA status and GHE's version-gating mean a plugin can fully implement `addBlockedBy` yet the connected instance not support it. `MethodNotFound` only models the first case. An explicit, instance-aware capability answer is needed, which is why the component SDK already has `host.capability.query -> { available, introducedIn }` (`plugin-sdk/src/types.ts:742-746`) as the FR-017 graceful version gate; the same shape fits here.
2. **The degrade must surface before the write, not as a caught error after a half-attempt.** The architecture's `POST /gates/:gateId/fix-issues` returns `422` when the capability is absent (architecture.md:147), and NFR-003's create-then-link atomicity requires knowing up front whether the link step can succeed, so the filer does not create an orphan issue it then cannot wire (the `link_pending` partial-state path, prd.md:156-158). A pre-flight capability check enables that; a post-hoc `MethodNotFound` catch does not.
3. **Consent is gated on declared capability.** NFR-001 requires these ops run "only through a declared, user-consented integration-plugin capability." Roubo's consent model is built on _declared_ manifest categories (`shared/plugin-consent-schema.ts`: `declaredCategories` / `allCategoriesAcknowledged`). A capability the user consents to must be declarable in the manifest, which means an explicit flag, not an implicit method-presence inference.

**The flags.** Define two booleans, surfaced on the manifest `capabilities` object (which exists for exactly this and is currently empty) and consumed by the `TrackerActionGateway`:

- `supportsCreateIssue: boolean`: the plugin implements `createIssue` for the connected instance.
- `supportsBlockingLinks: boolean`: the plugin implements `addBlockedBy` AND the connected instance exposes the blocking-link write (the GitHub GA / GHE version / Jira link-type-resolves condition from AC1). This flag is the one that may be **resolved at runtime** rather than statically, because its truth depends on the instance, not just the plugin build. The component SDK's `CapabilityQueryResult { available, introducedIn }` is the precedent shape for that runtime resolution.

A third op, close-gate, is not a new flag: the architecture already notes it can reuse the existing `applyTransition` to the done state when `allowedTransitions` exposes one (architecture.md:133-134), so it rides the existing transition capability rather than minting a new one.

**Degraded-mode behaviour (the explicit contract for "no silent no-op").** When the active tracker lacks a capability:

- **The gateway never calls the missing op.** It checks the flag first and, if false, returns a typed capability-absent outcome that the route maps to `422` (architecture.md:147), the filer surfaces as a clear operator message naming the tracker and the missing capability (for example: "This GitHub Enterprise instance does not support issue blocking links, so the fix issue cannot be wired to block the gate automatically. File it manually, or link it in the tracker."), and an audit-log entry records the attempt-and-degrade (NFR-001: an entry per action, including the refused one).
- **No silent no-op, ever.** The failure is loud and legible at the point of action, mirroring how Jira's `getSourceOptions` already logs and re-throws rather than swallowing (`plugin.ts:282-291`) and how the github family surfaces per-category fetch failures as visible warnings rather than dropping them (`ListIssuesWarning`).
- **Atomicity-aware degrade for create-then-link.** Because `supportsCreateIssue` can be true while `supportsBlockingLinks` is false (a tracker that can file but not link), the filer checks BOTH flags before creating. If linking is unsupported it tells the operator up front rather than creating an issue it then cannot wire; if linking is supported but the link call fails after a successful create, it returns the `link_pending` partial state with the created ref for a link-only retry (NFR-003, architecture.md:121-122). The capability flag prevents the predictable-up-front orphan; the retry path handles the unpredictable post-create failure.

This keeps every privileged op consent-gated (NFR-001), capability-flagged (NFR-005), and audit-logged through the existing `AuditLog` store (`server/services/audit-log.ts`), with the degrade always a clear surfaced message.

## Recommendation

**Adopt.** The cross-tracker create + blocking-link capability is feasible on all three trackers, with known, bounded gaps, and the GitHub-first phasing is the right route. Carry these decisions into the tracker-action gateway (#705) and the fix-issue filer (#706):

1. **Two new plugin ops, GitHub first.** `createIssue({ title, body, labels }) -> { ref, url, node_id? }` and `addBlockedBy({ blockedRef, blockerRef }) -> void` (architecture.md:130-132), added to the `PluginContract` and the `CONTRACT_METHODS` allowlist, wired in github-com first. GitHub create uses GraphQL `createIssue` (or REST `POST /issues`); GitHub blocking-link uses the issue-dependencies write mutation, the WRITE neighbour of the `blockedBy` READ already shipping in `github-fetchers.ts`.
2. **Two explicit capability flags, not the implicit `MethodNotFound` model.** `supportsCreateIssue` and `supportsBlockingLinks`, declared on the manifest `capabilities` object (currently empty, `schema/roubo-plugin.schema.json:53-57`) and consumed by the `TrackerActionGateway`. `supportsBlockingLinks` is resolved against the connected instance (GitHub GA status / GHE version / Jira link-type resolution), reusing the `CapabilityQueryResult { available, introducedIn }` shape the component SDK already defines.
3. **Loud degrade, never a silent no-op.** When a flag is false the gateway returns a typed capability-absent outcome (`422` at the route), surfaces a clear operator message naming the tracker and missing capability, and audit-logs the refused attempt (NFR-001, NFR-005). The filer checks both flags before creating so it never orphans an issue it cannot wire (NFR-003).
4. **Per-tracker write surface, settled:** GitHub: GraphQL/REST create + issue-dependencies mutation (probe GA). GHE: same surface once the plugin is write-wired, blocking-link version-gated by the GHE release. Jira: REST `POST /rest/api/2/issue` create + `POST /rest/api/2/issueLink` with the configured blocks-family link type.
5. **close-gate reuses `applyTransition`** to the done state rather than a new op/flag (architecture.md:133-134).

No host-API, manifest-schema, or plugin source change is made by this spike; it fixes the contract #705 and #706 implement.

## Open questions and follow-ups

- **GitHub blocking-link write GA status is date- and instance-dependent.** The issue-dependencies WRITE mutation has been maturing toward GA; the gateway must probe its availability on the connected instance at action time rather than assume it from the READ shipping today. This is an implementation detail for the #705 gateway slice (the capability-probe path), not a decision this spike needs to settle further. It should be tracked on #705.
- **GHE is write-unwired today.** `plugins/ghe/src/index.ts` wires no write methods at all, so GHE's `createIssue` / `addBlockedBy` are greenfield, and the blocking-link write is additionally version-gated by the GHE release. Bringing GHE to write parity (and the version probe) is deferred behind the GitHub-first phasing and belongs to a GHE-specific slice under #705, not this spike.
- **Jira link-type resolution.** The blocking-link write depends on the configured blocks-family link type existing on the instance (the user may have renamed it; `link-types.ts` already carries the configurable names). The gateway should validate the link type resolves before the write and degrade legibly if it does not; this is part of the Jira slice of #705.
- **Audit-log scope for integration-plugin writes.** The existing `AuditLog` (`server/services/audit-log.ts`) is wired today to the per-bench component-plugin broker, not to integration-plugin write RPCs (`pluginManager.invoke` is a plain pass-through with no audit hook, `plugin-manager.ts:1458-1515`). Routing the new create/link/close ops through the audit log (NFR-001) therefore needs a small host-side addition in the `TrackerActionGateway` path. This is in-scope for #705 and is flagged here so the gateway slice budgets for it; it is not a contract gap this spike resolves.
