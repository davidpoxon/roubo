# Decisions log: integration-plugins

Captured during structured product-flow stages. Each entry has a date, the question, the answer, a one-line rationale, and the stage that captured it.

- **2026-05-21** — Plugin kinds in scope this slug → integrations only. Rationale: keep scope tractable; the runtime is designed forward-compat for AI-agent and project-component plugins via FR-038. (captured during /product-interview)
- **2026-05-21** — GitHub.com re-implementation strategy → full rewrite as the bundled plugin (no legacy code path retained). Rationale: avoid carrying two GitHub flows; matches the cleanest end-state, accepts the rewrite-fidelity risk on `githubRequest`. (captured during /product-interview)
- **2026-05-21** — Migration of existing GitHub.com users → automatic + invisible on first launch. Rationale: no upgrade penalty for current users; FR-027 / NFR-009 handle the atomicity. (captured during /product-interview)
- **2026-05-21** — Write-back operations in scope → status transitions + assign/unassign only. Rationale: covers the bench workflows users actually want; commenting and PR-link write-back deferred. (captured during /product-interview)
- **2026-05-21** — Plugin runtime architecture → child Node process per plugin, JSON-RPC over stdio (recommended transport: `child_process.spawn` + `vscode-jsonrpc` framing per feasibility). Rationale: crash isolation, kill / restart without restarting Roubo. (captured during /product-interview)
- **2026-05-21** — Plugin language for this slug → Node-only. Rationale: smallest spec; non-Node plugins can be added in a follow-on slug without breaking compat. (captured during /product-interview)
- **2026-05-21** — Plugin discovery dirs → bundled `plugins/` and `~/.roubo/plugins/<plugin-id>/`. Rationale: avoids a third trust surface and avoids npm-style global module concerns. (captured during /product-interview)
- **2026-05-21** — Plugin choice lives in `roubo.yaml` `integration` block with field-level optionality, layered with a per-user override; per-field deep merge; arrays REPLACE. Rationale: small focused projects commit everything; large multi-team monorepos commit nothing; matches both ends of the spectrum without forcing one model. Array-replace per investigator recommendation; avoids "user can never remove a team default" surprises. (captured during /product-interview and feasibility)
- **2026-05-21** — Missing plugin on project load → prompt to install in one click. Rationale: lowest-friction onboarding for a teammate cloning a repo. (captured during /product-interview)
- **2026-05-21** — Plugin install sources → bundled, Git URL, local directory path. NOT tarball / zip. Rationale: covers bundled + community + local-dev workflows; tarball adds another verification pipeline for no clear win. (captured during /product-interview)
- **2026-05-21** — Plugin versioning in `roubo.yaml` → plugin id only (no semver pin or range). Rationale: users run whatever they have installed; plugins responsible for backwards compat within `hostApiVersion`. (captured during /product-interview)
- **2026-05-21** — Permission categories declared in manifest → four (network hosts, credential slots, filesystem, child-process spawning). Rationale: covers every escape hatch a plugin could use; child-process explicit because Jira ADF walkers / git invocations are realistic plugin needs. (captured during /product-interview)
- **2026-05-21** — Credential store → OS keyring via pure-JS shellout to platform CLIs (`security`, `secret-tool`, PowerShell). Rationale: honours no-native-modules constraint; cross-platform; feasibility spike A required. (captured during /product-interview and feasibility)
- **2026-05-21** — Self-signed TLS → per-plugin opt-in, off by default, warning shown. Rationale: matches real on-prem GHE / Jira deployments without weakening the global security boundary. (captured during /product-interview)
- **2026-05-21** — Native modules → hard constraint, not a preference. Rationale: avoids native-build complexity on user machines; forces the keyring shellout approach. (captured during /product-interview)
- **2026-05-21** — GitHub.com auth → keep the Roubo OAuth app. Rationale: existing users don't re-authenticate; deep-link callback retained. (captured during /product-interview)
- **2026-05-21** — GHE auth → PAT + instance URL. Rationale: simplest enterprise onboarding; no per-instance OAuth-app registration. (captured during /product-interview)
- **2026-05-21** — Jira self-hosted auth → PAT only (Data Center 8.14+). Rationale: targets modern Data Center; older Jira Server out of validated scope. (captured during /product-interview)
- **2026-05-21** — Identity for write-back → derive from credentials at config time via `plugin.getCurrentUser()`. Rationale: zero user typing of usernames; one identity per (project, integration) is enough. (captured during /product-interview)
- **2026-05-21** — Plugins management UI → new top-level Plugins settings page. Rationale: clearest information architecture; Connections / GitHub-connect tile is conceptually narrower. (captured during /product-interview)
- **2026-05-21** — Per-project integration config UI → "Issue source" tile on the project detail page. Rationale: matches the GitHub-Project picker pattern users already know. (captured during /product-interview)
- **2026-05-21** — Source picker rendering → declarative shapes (`multi-list`, `categorized-multi-list`), host renders. Rationale: avoids the plugin-supplied React surface entirely; matches the security model. (captured during /product-interview)
- **2026-05-21** — Status-transition triggers → never auto-fire. Roubo offers the actual next states from `allowedTransitions`, not a generic "Done." Rationale: avoids surprising users by misreading a Jira workflow; explicit is safer than implicit. (captured during /product-interview)
- **2026-05-21** — Issue state model → plugin returns `currentState` + `allowedTransitions` in the normalized issue. Rationale: keeps Roubo workflow-generic; plugin owns the state-machine awareness. (captured during /product-interview)
- **2026-05-21** — Blocks / blocked-by enforcement → soft warning, not hard block. Rationale: trusts the user; reduces friction. NOTE: this is a deliberate relaxation of current Roubo behaviour at `server/services/issue-assignment.ts:102`; must ship in lockstep with `client/src/components/IssuePickerModal.tsx:91`. (captured during /product-interview)
- **2026-05-21** — Issue hierarchies (epics, parents, subtasks) → out of scope this slug. Rationale: keeps the normalized contract narrow; can be added without a host-API major bump. (captured during /product-interview)
- **2026-05-21** — Polling strategy → on-demand + UI events. No background timer. Rationale: matches current behaviour; lowest API load; users control refresh. (captured during /product-interview)
- **2026-05-21** — Migration target → per-user override file, NOT the committed `roubo.yaml`. Rationale: migration is per-user; we never auto-commit changes to a checked-in file. (captured during /product-interview)
- **2026-05-21** — Plugin restart policy → 3 restarts in 5 minutes, then mark errored. Rationale: bounded; protects against perpetual-retry hiding a real problem. (captured during /product-interview)
- **2026-05-21** — Performance model → paginated `listIssues` with default page size 50, plugin-configurable. Rationale: replaces a "fetch-all" time budget with a UX-aligned model; matches `useInfiniteQuery` shape. (captured during /product-interview)
- **2026-05-21** — Forward-compat investment → design intentionally + verify with a one-page paper sketch before `hostApiVersion` 1.0.0 freeze. Rationale: cheap insurance against a major bump on the next two plugin slugs. (captured during /product-interview)
- **2026-05-21** — Test connection UX → explicit button calling `plugin.validateConfig()`. Rationale: lets users probe-without-committing; clearer error feedback than save-time-only validation. (captured during /product-interview)
- **2026-05-21** — Leading indicators → owner dogfoods bundled GHE + Jira for 2 weeks before public release; zero migration regressions. External alpha testers and third-party-plugin-in-a-day NOT picked as gates. Rationale: minimal viable gating; external tests are nice but not blocking. (captured during /product-interview)
- **2026-05-21** — Lagging indicators → zero P0 security incidents in 6 months; integration-config support volume under 10%. Adoption % and community-plugin count NOT picked as gates. Rationale: success is "safely shipped, low support burden"; adoption metrics deferred until telemetry exists. (captured during /product-interview)
- **2026-05-21** — Recommended transport library → `child_process.spawn` + `vscode-jsonrpc` framing (NOT `child_process.fork`). Rationale: `fork`'s Node IPC channel does not work well for plugins installed under `~/.roubo/plugins/<id>/` per feasibility investigation. (captured during /product-feasibility)
- **2026-05-21** — Array merge semantics for the `roubo.yaml` + user override deep-merge → REPLACE (not concat). Rationale: investigator recommendation; avoids "user can never remove a team-committed source" surprises. (captured during /product-prd checkpoint)

Affects keyring matrix (macOS + Ubuntu desktop + Ubuntu headless only; no PowerShell / `cmdkey` / DPAPI), CI matrix (no Windows runner), and integration test platforms. Earlier sections of `feasibility.md` and `architecture.md` that discussed Windows are superseded; the working assumption is now macOS + Linux only. (captured during /product-architecture checkpoint)
