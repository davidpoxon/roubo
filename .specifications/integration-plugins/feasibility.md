# Feasibility: Integration plugins (extensible plugin system, first use case: issue-source integrations)

> Slug: `integration-plugins` · Investigated: 2026-05-21
>
> **Superseded note (2026-05-21):** the user clarified after this report was written that Roubo does not support Windows as a host platform (licensing). All references below to Windows, PowerShell, `cmdkey`, DPAPI, or Windows CI runners are superseded by `architecture.md` and the constraint in `context.md`. The keyring matrix is now macOS + Ubuntu desktop + Ubuntu headless only.

## Recommendation

**build-with-spike** — the runtime, manifest loader, JSON-RPC transport, route re-routing, and migration all have direct prior art in this codebase and are buildable. Two areas need spikes before host-API freeze: (1) a pure-JS OS-keyring shellout that works on macOS, Ubuntu desktop, and Ubuntu headless (Windows is out of scope per the superseded note above), and (2) a faithful re-implementation of the existing GitHub.com ETag / rate-limit / GraphQL-batching behaviour inside the new plugin process across a JSON-RPC boundary.

## Prior art

- **`githubRequest` central helper (ETag store, exponential backoff, 429/403/secondary-rate-limit handling, stable cache key)** — `server/services/github.ts:255`. Must be lifted verbatim into the bundled github.com plugin. Note: the `getOctokit()` factory at `server/services/github.ts:101` reads from `auth.json` directly; in the new world the plugin reads its token from `host.credentials.get(...)`.
- **GraphQL batching for blocking relationships** — `server/services/github.ts:476` (`buildBlockingQuery`) and `:516` (`fetchBlockingRelationships`). 20-issue alias batches, 3-level recursion, OPEN-only flattening. Reusable as-is inside the plugin.
- **GitHub Projects v2 pagination (10-page cap, org-or-user query fallback, single-select status column)** — `server/services/github.ts:840`. The MAX_PAGES constant is the closest thing the codebase has to "paginated `listIssues`"; the new normalized contract should expose this as a real cursor instead of a hard cap.
- **GitHub issue-types fetcher** — `server/services/github.ts:960` (`fetchIssueTypes`). Drives `/api/projects/:projectId/issue-types`. Replaceable by `plugin.listIssueTypes()` (a method the SDK should add) with the same `{ configured, reason, types }` envelope.
- **CrossReferenced PR seeding** — `server/services/github.ts:618` (`fetchLinkedPullRequests`). Drives `AssignedIssue.linkedPullRequests` and seeds `BenchWorkUnit.pullRequest`. GitHub-specific; Jira plugin returns `[]`. Already best-effort, so dropping it under Jira is safe.
- **Open-PR-by-branch with ETag-aware return** — `server/services/github.ts:1061` (`fetchOpenPullRequestByBranch`). The `{ notModified, pr }` shape is consumed by `server/services/pr-sync.ts:129`. Out of scope for Jira per context (meta-repo PR sync against non-GitHub is explicit out), so this stays GitHub-only; the github.com plugin re-exports it via an SDK helper (or the pr-sync stays bound to the github.com plugin and short-circuits for projects whose active integration is not github.com).
- **OAuth deep-link handler** — `electron/src/main.ts` plus `electron/src/main.test.ts:127`. The handler explicitly matches `roubo://oauth/github/callback`; it stays put and the bundled github.com plugin owns the token slot it writes to.
- **GitHub OAuth state machine, token exchange, scopes, status cache** — `server/services/github-auth.ts:1` (file-level), with `saveCredentials` at `:132` writing to `auth.json` and `deleteCredentials` at `:145`. The persistence call site is the migration's source of truth; the network-side code is the bundled github.com plugin's `connect()` / `validateConfig()` implementation.
- **Hard-block check on bench creation** — `server/services/issue-assignment.ts:102` (`checkIssueDependencies`). This is the *only* hard block site; it throws `ServiceError(409, "Issue is blocked by unresolved dependencies", { blockedBy })`. Under the new soft-block rule, this function becomes a passthrough that attaches the blocker list to the response so the UI banner can render. The `enforceIssueDependencies` toggle (`server/services/project-registry.ts:215`) and the UI's disabled-row treatment at `client/src/components/IssuePickerModal.tsx:91` are the two places that drive the existing hard-block UX; both have to flip to "warn, don't block."
- **Blueprint manager: manifest-with-frontmatter, FS watcher, debounced invalidation, TTL safety net** — `server/services/blueprint-manager.ts:65` (`loadBlueprintFile`), `:372` (`debounceReload`), `:384` (`startWatchers`), `:405` (`startAppBlueprintsWatcher`). This is the direct template for the plugin manifest loader: same YAML-frontmatter shape, same per-dir watcher pattern, same debounce-then-invalidate. Reuse the watcher map idiom for the two plugin discovery directories.
- **Two-tier discovery with override** — `server/services/blueprint-manager.ts:179` (`resolveBlueprintsForProject`). The bundled-then-user-installed precedence translates directly to plugins: bundled `plugins/` first, `~/.roubo/plugins/<id>/` second, with collisions on `pluginId` resolved by "bundled wins" (or the inverse — call this in PRD; the blueprint manager picks user-overrides-bundled, which is the opposite of what we'd typically want for plugins where bundled is the trusted set).
- **Child-process lifecycle with logs, treeKill SIGTERM-then-SIGKILL** — `server/services/process-manager.ts:1`. Direct template for the plugin supervisor. The 5000ms grace window, ring-buffered logs (5000 lines), and `stopAllProcesses` for shutdown all transfer. Add: a per-plugin restart-budget tracker (3 attempts / 5 min) layered on top of `proc.on('close', ...)`.
- **Long-running managed child with abort and per-line buffer** — `server/services/inspection-runner.ts:1`. Closer in shape to a plugin process than `process-manager.ts` because of the bounded-output ring buffer and abort signalling.
- **Subprocess + timeout + captured I/O** — `server/services/exec.ts:32` (`runCommand`). Relevant for the keyring shellouts (`security`, `secret-tool`, `powershell`). Returns `{ code, stdout, stderr }` with a SIGTERM timeout. Reusable as-is, but: it currently passes `cleanEnv()`; the keyring shellout needs `DBUS_SESSION_BUS_ADDRESS` and on Linux server may need `secret-tool` to be unlocked, so `cleanEnv` may need a passthrough for those vars or the keyring helper bypasses `runCommand`.
- **roubo.yaml schema with field-level optionality, additive non-breaking changes already in flight** — `shared/config-schema.ts:235` (`RouboConfigSchema`) and the parallel JSON Schema at `schema/roubo-config.schema.json`. The schema is already `.strict()`, which means an unknown top-level `integration` block would be rejected — adding the block is a real schema migration, not a no-op.
- **Atomic file writes** — `server/services/state.ts:49` (`atomicWrite`). Tmp file + rename, with explicit mode. Sufficient for the per-user override file and the state-version-marker bump. Not sufficient on its own for *cross-file* atomicity (auth.json delete + override write + state version bump): we need a 2-phase commit pattern wrapping multiple `atomicWrite` calls plus the unlink, with a recovery path keyed off the state-file version marker.
- **Per-project per-file state storage** — `server/services/state.ts:206` (`resolvePermissionsPath`). The `~/.roubo/permissions/<projectId>.json` pattern with traversal-safe path resolution is the direct template for the per-user override file at `~/.roubo/integrations/<projectId>.json` (or similar).
- **`UserPreferences` partial-merge defaults** — `server/services/state.ts:162` (`loadSettings`). Spreads defaults over the read file. This is shallow — a deep merge utility does not yet exist in the repo and will need to be added (or zod's `deepPartial` + a custom merger).
- **PR-sync as a polling tick** — `server/services/auto-clear.ts:32` (`start`). 30-second `setInterval` orchestrating PR sync + classification. No precedent for "no background timer" issue polling, which is what the new plugin model wants (on-demand + UI events only). The new shape is closer to React Query staleTime than to `auto-clear.ts`.
- **Octokit dependency surface** — `server/package.json:32` (`octokit: 5.0.5`). The Octokit dependency is currently a host-level concern; in the new world it moves into `plugins/github-com/package.json` and the host package loses the direct dep (or keeps it for the github.com plugin only via a workspace).
- **`ajv` is vestigial** — confirmed: no `import.*ajv` site exists across the codebase; only declared in `package.json:45` and `server/package.json:25`. Safe to drop in this slug. Validation lives in `shared/config-schema.ts` via zod.

## Capability gaps

- **JSON-RPC over stdio transport** — no precedent in the repo. Recommend `child_process.spawn` (not `fork` — `fork` requires the child to be a Node script run from the same install and creates a Node IPC channel that does not work for "user-installed module in `~/.roubo/plugins/<id>/`" the same way) + Content-Length-framed JSON-RPC (the same wire format as LSP / DAP). Library: `vscode-jsonrpc` is widely used, has zero native deps, and uses the framed-stream protocol; it does not impose vscode-specific types. `jsonrpc-lite` is lighter but message framing is hand-rolled. Candidate approach: `vscode-jsonrpc` with `StreamMessageReader` / `StreamMessageWriter` bound to `proc.stdout` / `proc.stdin`. Reject `fork` for portability reasons.
- **Plugin manifest format (`roubo-plugin.yaml`) loader and zod schema** — gap. Adjacent prior art is `loadBlueprintFile` (frontmatter loader, size limits, frontmatter required fields). Candidate approach: a top-of-file YAML document at `<pluginDir>/roubo-plugin.yaml`, parsed by `yaml` (already a dep at `package.json:54`), validated by a zod schema that lives in `shared/plugin-manifest.ts` (new file) and reused by both host and SDK. JSON Schema artifact emitted to `schema/roubo-plugin.schema.json` for IDE autocomplete — but note: zod-to-json-schema is not a current dep; either add `zod-to-json-schema` or hand-author the JSON Schema in parallel with the zod (the codebase already does this for `roubo.yaml`).
- **Host-provided permission enforcement (network allowlist, credential slot scopes, FS confinement, child-process spawning)** — gap. No precedent for sandboxing inside this repo. The four categories map to: network = host owns the HTTP client (see `host.fetch` below), credentials = host-only RPC method, FS = the plugin only gets a working directory passed in over RPC (no enforcement at the OS level), child-process = the plugin requests `host.spawn()` via RPC rather than calling `spawn()` directly. The honest framing: this is *cooperative* enforcement, not adversarial sandboxing — a malicious plugin can still call Node APIs directly. Permission enforcement is best-effort and tied to user-accepted permissions plus the install-source URL the user vetted. Mark this clearly in the docs.
- **`host.fetch` HTTP client with allowlist + system proxy + per-plugin self-signed-TLS opt-in** — gap. Recommend `undici` (built-in to Node 24, no install). `undici.Dispatcher` plus `EnvHttpProxyAgent` covers system proxy (`HTTPS_PROXY`, `NO_PROXY`). For self-signed: a per-plugin `Agent({ connect: { rejectUnauthorized: false } })`; honour the manifest field `allowSelfSignedTls: true` only after the user has explicitly enabled it in the per-plugin configure dialog. Note: the plugin sends a request over RPC, host performs the I/O with the allowlist applied, host returns the response body and status code. This rules out plugins streaming large bodies efficiently — fine for this slug's use cases (issue lists), worth noting for future plugin kinds (project components might need streaming).
- **OS keyring via pure-JS shellouts** — gap, named in the context as Spike A. Concrete shellouts:
  - macOS: `security add-generic-password -a <account> -s <service> -w <password> -U` and `security find-generic-password -a <account> -s <service> -w`. Available out-of-the-box.
  - Linux desktop: `secret-tool store --label='...' service <service> account <account>` reading from stdin (so the password never appears in argv). `secret-tool lookup service <service> account <account>` to read. Provided by `libsecret-tools` package; on Ubuntu Desktop it ships, on Ubuntu headless / server it does NOT and needs `apt install libsecret-tools gnome-keyring`, plus an unlocked keyring. This is the hard part of Spike A.
  - Windows 11: PowerShell + Windows Credential Manager. The native API is `CredWrite`/`CredRead`; without native modules, the workable approach is `powershell -Command "..."` invoking `New-Object Windows.Security.Credentials.PasswordVault` via UWP COM, OR shelling to `cmdkey.exe /generic:<target> /user:<account> /pass:<secret>` (but `cmdkey` does not return the secret to stdout when read, only metadata — so it's write-only and we need a different read path). Realistic candidate: a small PowerShell snippet using `Microsoft.Web.Services3` or the `PSCredentialManager` module, but neither ships by default. **Honest assessment: pure-JS keyring on Windows 11 without any installed module is hard, and the spike must produce a working read-after-write loop before host-API freeze.** Avoid pure-JS wrapper libs like `keytar-pure` or `node-keyring`; many of them actually re-shell to the same CLIs or depend on native modules.
  - Linux server (headless): if `gnome-keyring-daemon` is not running, `secret-tool` fails. Documented fallback path: a passphrase-encrypted file at `~/.roubo/credentials/<pluginId>.enc` with a master passphrase prompted at first launch. This is a meaningful UX hit and changes the constraint described in the context ("OS keyring only") — surface as an Unknown if the spike confirms it.
- **Deep merge of `roubo.yaml`-`integration` block + per-user override** — gap. The repo only has shallow merge (`server/services/state.ts:162` for settings). New utility needed. Array semantics: for `sources.boards: [...]`, the context says deep-merge but doesn't pick replace-vs-concat — recommend **replace** at the array level (user-override entirely replaces committed array; matches the spirit of "user controls their own view") and document this loudly. Edge case: an empty array in the override means "explicitly none," not "fall through" — null vs `[]` semantics needs to be a zod refinement.
- **Per-user override file location** — gap. Recommend `~/.roubo/integrations/<projectId>.json` (matches the `~/.roubo/permissions/<projectId>.json` pattern with same path-traversal guard) versioned with a small `{ schemaVersion: 1, integration: {...} }` envelope. Stored via `atomicWrite`.
- **State-file version marker for migration rollback** — gap. `PersistedState` in `shared/types.ts:432` has no version field. Add `schemaVersion: number` (default 0, migration target 1). Migration writes the new value last; on partial failure, presence of `schemaVersion: 0` after a crash signals "migration incomplete, rerun." Note: `loadState` at `server/services/state.ts:91` returns `{ benches: [] }` for missing files — same defaulting must apply for missing `schemaVersion`.
- **`integrationId` / `externalId` on `PersistedBench` + `BenchWorkUnit`** — gap. `shared/types.ts:401` (`PersistedBench`) and `shared/types.ts:646` (`AssignedIssue.number`) are numeric and GitHub-shaped. Adapter: at load time, default `integrationId = "github-com"` and `externalId = String(assignedIssue.number)` for benches with no `integrationId` field. New `AssignedIssue` shape: `{ integrationId, externalId, title, externalUrl, currentState?, allowedTransitions?, linkedPullRequests? }`. Downstream consumers: `server/services/pr-sync.ts:129` and `server/services/auto-clear.ts:51` only run for github.com benches (gated on `integrationId === "github-com"`).
- **Pagination threaded through to the UI** — gap. `useProjectItems` at `client/src/hooks/useProjectItems.ts:4` returns all items via React Query with a single `queryFn`. New shape: `useInfiniteQuery` with `getNextPageParam` reading a cursor from the response. The server route `/api/projects/:projectId/issues` (`server/routes/issues.ts:27`) accepts `labels` / `search` query params and returns the full array; new shape returns `{ items, nextCursor }`. UI components (`IssueQueuePanel.tsx:47`, `IssuePickerModal.tsx:131`) consume the `.items` array directly — they need an explicit "load more" trigger or auto-paginate-on-scroll.
- **Plugin Test Connection (`plugin.validateConfig(config)`) round-trip** — gap. The plugin contract gains the method; the host adds an inline-result API endpoint (something like `POST /api/projects/:projectId/integration/test`) that proxies to the active plugin's `validateConfig` and returns a structured success/error envelope. UI: a new button on the Issue source tile; React Aria `<Button>` with a pending state.
- **Source picker shapes (`multi-list`, `categorized-multi-list`)** — gap. `MultiSelect.tsx` exists (`client/src/components/MultiSelect.tsx`) and is a starting point for `multi-list`. `categorized-multi-list` (Jira tabs of Boards / Epics / Filters) does not exist; React Aria's `Tabs` plus the existing `MultiSelect` should compose. Plugins return `{ shape: "multi-list", items: [...] }` or `{ shape: "categorized-multi-list", categories: [...] }`; host renders.
- **`getCurrentUser()` + identity capture** — gap. New plugin contract method. Host calls it at config-time, stores the returned external id on the project's persisted integration config (`~/.roubo/integrations/<projectId>.json` per-user override, so identity is per-user as it must be). On `assignIssue(externalId, userId?)`, the host passes the captured identity. Risk: GHE returns `login`, Jira returns `accountId` for Cloud or `key` for DC — the plugin's `getCurrentUser()` return is opaque to the host except as an identity blob.
- **Status transitions (`applyTransition`, `getAvailableTransitions`)** — gap. The current GitHub flow has no concept of state transitions (issues are open/closed). Bundled github.com plugin's `applyTransition` will map the strings `"close"` and `"reopen"` to the REST `PATCH /repos/{owner}/{repo}/issues/{issue_number}` `state` field. Bundled GHE plugin: same. Bundled Jira plugin: `GET /rest/api/2/issue/{key}/transitions` for available transitions, `POST /rest/api/2/issue/{key}/transitions` to apply. New normalized field `currentState` + `allowedTransitions` lives in the issue contract.
- **Assign / unassign (`assignIssue(externalId, userId?)`, `unassignIssue(externalId)`)** — gap. GitHub: `POST /repos/{owner}/{repo}/issues/{issue_number}/assignees`. GHE: same. Jira: `PUT /rest/api/2/issue/{key}/assignee` with body `{ accountId | name }`. UI: an in-bench control on the bench detail / queue panel; `client/src/components/BenchDetail.tsx` is the natural mount point.
- **Plugin SDK package** — gap. New workspace `sdk/` (or under `plugins/sdk/`). Exports: `definePlugin({ ... })`, `host.fetch`, `host.credentials`, `host.logger`. The plugin process imports from `@roubo/plugin-sdk` and is given a `host` proxy bound to RPC. Versioned independently with `hostApiVersion` range declared in `roubo-plugin.yaml`.
- **Plugin install from Git URL** — gap. Candidate: shell out to `git clone <url> ~/.roubo/plugins/<pluginId>/`. Roubo already uses `git` shellouts elsewhere (`server/services/git-helpers.ts`, `git-state.ts`). After clone, read `roubo-plugin.yaml`, validate, show permissions dialog, accept or roll back via `rm -rf` of the cloned dir.
- **Jira ADF (Atlassian Document Format) → markdown** — gap. The Jira REST API for DC 8.14+ returns issue body as ADF JSON for new content. Hand-rolled walker is straightforward (tree-walk of node types: paragraph, text, hardBreak, bulletList, etc.); `@atlaskit/adf-utils` exists but is a heavy and Atlassian-centric dep. Recommend hand-rolled walker scoped to the node types Jira DC actually emits. If body comes back as plain text (DC option), no conversion needed.
- **GraphQL batching in Jira plugin** — n/a. Jira REST has no batching primitive equivalent. Per-issue HTTP, with a concurrency limiter (e.g. 5 in-flight). The plugin owns this.

## Risks

### Technical

- **JSON-RPC framing inside a packaged Electron app on Windows.** stdio is line-buffered differently across platforms; Content-Length-framed messages with `vscode-jsonrpc` handle this, but the spike must verify it under packaged Roubo on Windows 11. Resolution: write a 50-line reference plugin that echoes requests; verify under `electron:make`.
- **Re-implementing `githubRequest` across an RPC boundary preserves ETag/304 semantics.** The cache lives in the plugin process; the host sees only normalized payloads. The host's HTTP client (`host.fetch`) must surface response headers (specifically `ETag` and `Retry-After`) to the plugin or the plugin's ETag store is useless. Resolution: `host.fetch` returns `{ status, headers, body }`, plugin owns the ETag store. Note: this means `host.fetch` is not just a sandboxed wrapper but a transparent passthrough of headers — re-confirm that fits the "host enforces TLS opt-in and allowlist" model (it does).
- **Plugin process zombie on Roubo crash.** Roubo crashes → child plugin processes orphan → they keep TCP connections to GitHub / Jira open. `process-manager.ts:153` (`stopAllProcesses`) handles graceful shutdown via beforeExit; SIGKILL on parent leaves children behind. Resolution: use `detached: false` + `proc.unref() === never` (children die with parent on most platforms when the parent exits cleanly); on Windows, the job-object trick (`spawn(..., { windowsHide: true })` and parent-owned job) is required. Spike or accept the orphan risk and document.
- **`fetchIssueTypes` GraphQL query failure mode.** `server/services/github.ts:944` issues an `issueTypes` field on `repository` — GitHub returns a schema-level error for orgs without the feature enabled, not a graceful empty array. Migration risk: a GHE instance running an older GitHub schema (pre-Issue Types) returns a hard error. Resolution: the GHE plugin's `listIssueTypes` catches schema errors and returns `{ configured: false, reason: "none-defined" }` — same fallback already in the existing code at `server/services/github.ts:988`.

### Integration

- **Octokit dep movement.** `octokit` is currently a top-level server dep; it now lives only in the bundled github.com plugin. The host loses `Octokit` as an importable symbol; any test fixture or helper that imports from `octokit` directly needs to move. Resolution: grep for `import.*octokit` (already done — 9 files, all in `server/services/github.ts`, `server/services/github-error.ts`, `server/routes/auth.ts`, and their tests) and migrate them into the github.com plugin package.
- **`roubo://oauth/github/callback` is hardcoded in Electron and in github-auth.** No abstraction layer. Resolution: keep it hardcoded as a github.com-plugin-specific deep link. Out of scope is generalizing this for arbitrary plugins; if a future plugin needs a deep link, it gets its own scheme path under `roubo://oauth/<pluginId>/callback` and the dispatcher in `electron/src/main.ts` routes by prefix. Document the convention now.
- **`.strict()` zod root means the `integration` block must land before any user can write `integration:` in roubo.yaml.** No partial rollout window. Resolution: ship schema + parser in one PR; the migration handles the per-user-override side independently of the schema.
- **Existing `useProjectItems` query key shape.** React Query keys are `["project-items", projectId, projectNumber]` at `useProjectItems.ts:6`. Multiple components consume this. A rename to `["issues", projectId, integrationId, ...]` ripples through `useRefreshProjectItems` and any cache-invalidation site. Resolution: do the rename atomically with the route change; the test file at `client/src/hooks/useProjectItems.test.tsx` will catch most regressions.

### Non-functional

- **Soft-block migration changes UX expectations.** Roubo users today see blocked issues disabled in `IssuePickerModal`. Switching to a banner is a behavioural change. Existing tests at `server/services/issue-assignment.test.ts` and `client/src/components/IssuePickerModal.test.tsx` assert today's behaviour. Resolution: a section in the PRD + test plan explicitly covering the behavioural flip; one-time banner on first launch post-migration if any project had `enforceIssueDependencies: true` previously.
- **Per-plugin TLS rejection toggle.** `host.fetch` is per-plugin; the toggle lives in the plugin's stored config (per-user override), not in a global setting. A user who enables it for one plugin does not enable it for another. UI: a labelled "Allow self-signed certificates" checkbox in the plugin's configure dialog with an inline warning. Off by default. Risk: a confused user accepts the warning blindly. Mitigation: log the toggle change to `~/.roubo/plugins/<id>/logs/` so an audit trail exists.
- **Plugin restart budget hides reliability problems.** 3 restarts / 5 min is generous; a flapping plugin that recovers on attempt 3 every 5 minutes produces a degraded UX with no signal. Resolution: surface every restart in the Plugins page (last-5-restart timeline + most recent error) so the user can see it.
- **Pagination cache invalidation on poll-refresh.** If page 1 returns a new top-of-list item, an infinite-query cursor that's already on page 3 has a hole. React Query's `invalidateQueries(["issues", projectId, integrationId])` resets the whole infinite query; the UI must reset to page 1 on a manual refresh. Acceptable but document.
- **No documented "fetch everything" budget.** Context explicitly drops this. UI must never block on pagination; the picker shows results as pages arrive. Validate during build that the loading state in `IssuePickerModal` doesn't gate on `isFetchingNextPage`.

### Data

- **Atomic migration touches three persistence sites + one delete.** auth.json → keyring write, per-user override write, state.json schemaVersion bump, auth.json unlink. Each `atomicWrite` is itself atomic, but the sequence is not. Resolution: state.json is the commit marker; written *last* with the version bump. If a crash happens before state.json is bumped, the migration re-runs on next launch (it re-detects auth.json or partial override and finishes). The unlink of auth.json happens after state.json is bumped to schemaVersion 1, so an unlink-failed crash leaves auth.json sitting next to a bumped state and the next-boot logic must handle "schemaVersion 1 + auth.json still present" by re-attempting the unlink.
- **`PersistedBench.assignedIssue.number: number`** — `shared/types.ts:646`. Changing to a string `externalId` would break old state files. Resolution: keep `number` as legacy field; add `externalId: string` and `integrationId: string` as new fields with load-time defaulting (`integrationId = "github-com"`, `externalId = String(number)`). After 1 release cycle, evaluate dropping `number`. The migration itself does not need to rewrite state.json benches — they're defaulted in-memory.
- **Per-user override file is per-project.** Multiple projects → multiple files. Path-traversal safety must mirror `resolvePermissionsPath` (`server/services/state.ts:206`). If a project is unregistered, its override file is not orphan-cleaned today; permission files have the same issue. Acceptable — out of scope.
- **No regulatory regime applies to Roubo (local dev tool).** GDPR / FedRAMP / SOC2 are not in play. The opaque `raw` field is the only personal-data risk; context explicitly says plugins MUST NOT include PII in `raw` unless functionally required, and Roubo does not persist `raw` beyond an active bench. Acceptable.
- **Plugin uninstall while a project still references it.** Context decision: same "missing plugin → prompt to install" flow as for a teammate cloning a repo. Implementation: project load tolerates a missing-plugin state; the project's Issue source tile shows "Plugin not installed: <pluginId>. Install? / Switch integration?". No automatic config rewrite.

## Unknowns

- **Spike A outcome:** does a pure-JS-shellout keyring work on (i) clean macOS, (ii) clean Ubuntu desktop, (iii) clean Ubuntu server / headless, (iv) Windows 11? The Windows 11 path and the Linux-server path are the two that could force a fallback to passphrase-encrypted on-disk credentials. If either falls back, the "no plaintext on disk" constraint in context softens — surface to user.
- **`hostApiVersion` semver policy after 1.0.0.** Context defers to architecture. The paper sketch for AI-agent and project-component plugins is the gating artifact: if those plugin kinds require new permission categories or new RPC methods that cannot be additive, host-API 1.0.0 is the wrong place to freeze. Resolution path: the paper sketch is produced before host-API freeze and reviewed against the proposed permission vocabulary.
- **Array semantics on deep merge.** Recommended above: replace, not concat. Needs an explicit decision in PRD because "deep merge" alone is ambiguous.
- **Plugin lifecycle on config change.** Context Open Question #1. Recommend: restart on credential or instance-URL change (safety: token rotation); live-reload on sources change (cheap: just a refetch). Defer to architecture.
- **`Test connection` during migration.** Context Open Question #3. Recommend: auto-run as part of migration verification — the user already had a working token; a failure here is signal, not noise. If it fails, surface inline with the migration banner.
- **`listSourceCandidates` paging.** Context Open Question #4. Recommend: always-all for v1, paginate later. The shape needs to support paging later without a breaking change → include an optional `nextCursor` field in the response that v1 plugins return as `undefined`.
- **In-app plugin logs surface.** Context Open Question #5. Recommend: ship the in-app surface; per-plugin logs are already on the Plugins page roadmap.
- **Failure surface for `applyTransition` permission gaps.** Context Open Question #7. Recommend: plugin returns a structured `{ kind: "permission-denied", message, allowedTransitions: [...] }` error; host surfaces inline next to the dropdown.
- **Forward-compat for AI-agent and project-component plugins.** The paper sketch determines whether the manifest permission vocabulary is sufficient. Today's four categories (network / credentials / FS / child-process) plausibly cover an AI-agent plugin (spawn a CLI, read/write workspace files, no network if the CLI tunnels through its own auth). A project-component plugin (database, dev-server) needs *port allocation* as a permission category, and possibly *docker / compose* as a category — neither is in today's set. Resolution: add `ports` and `docker` as two more permission categories now if the paper sketch surfaces them, OR leave a permission-category extension point so a future host-API minor version can add them.
- **Octokit version lock-in.** The bundled github.com plugin keeps Octokit 5.0.5. If Octokit ever has a security bump, the plugin update has to ship; Renovate already manages this. No action needed beyond keeping the plugin in the Renovate scope.

## Top risks (priority order)

1. **OS keyring without native modules** on Windows 11 and Linux server/headless. Spike A must complete with a working read-after-write loop on all four target environments before host-API freeze. If either Windows or Linux-server forces a passphrase-encrypted-on-disk fallback, the "OS keyring only" constraint in context softens.
2. **Faithful re-implementation of `githubRequest` across the RPC boundary.** ETag caching, 30s TTL, 429 / 403 / secondary-rate-limit backoff with the existing `computeBackoffMs` table, and GraphQL batching for blocking relationships all need to live inside the github.com plugin. `host.fetch` must return raw response headers so the plugin can drive its own ETag store. Mismatched semantics here regress every existing GitHub user.
3. **Atomic three-write-plus-unlink migration.** state.json acts as the commit marker; partial-fail recovery is keyed off that. Must be idempotent and tested with crash injection in the spike.
4. **Soft-block behavioural change.** Existing tests assert hard-block on `enforceIssueDependencies: true`. The behavioural flip is small in code but visible to every existing user; needs a one-time banner and updated tests.
5. **Plugin process orphan on parent crash.** Especially on Windows. Use job-objects on Windows, document the Linux/Mac SIGTERM cascade.
6. **`.strict()` zod schema means schema + parser + migration must ship in one PR.** No partial rollout for the `integration` block in roubo.yaml. Coordinate the JSON Schema artifact in lockstep with the zod.
