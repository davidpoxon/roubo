# Spike 599: Which OS-level isolation backend enforces a component plugin's declared permissions in v2 on a macOS-primary Node host, and what can it honestly guarantee?

**Status:** Resolved · **Issue:** #599 · **Class:** decision · **Resolves:** feasibility risk 2 (no off-the-shelf OS sandbox for a headless Node process on macOS; "safe by default" only approximable) and architecture.md Open question "The v2 isolation backend ... resolved by the sandboxing spike" · **Implements:** FR-018, NFR-001, US-009 · **Verified by:** none (research spike) · **Gates:** Phase 2 F2.3 PluginIsolationSandbox (also the v2 PermissionEnforcer slice it pairs with) · **Recommendation:** adopt

## Objective and method

Select the backend the v2 `PluginIsolationSandbox` Component (architecture.md Components table; PRD FR-018) will use to constrain a component plugin so it cannot exceed its declared permissions, on a macOS-primary Node host, and state honestly what that backend can guarantee versus only approximate.

This is a decide-and-document spike. Implementing the sandbox is out of scope (the v2 F2.3 `PluginIsolationSandbox` slice owns the build; architecture.md "Out of scope": "v2 sandboxing implementation"). The advisory v1 permission-declaration and consent model is also out of scope (F1.14 / NFR-001 v1 target; the `PermissionConsentUI` Component). This spike chooses the v2 enforcement backend and pins what F2.3 implements against; it writes no `server/` / `client/` / `shared/` code. The deliverable is this findings doc.

Method: take the direction the feasibility report and architecture already pre-committed (host-owns lifecycle, with the `HostComponentBroker` as the single privileged choke-point; the `PluginIsolationSandbox` explicitly "not Docker-locked"), evaluate each candidate backend the issue and FR-018 enumerate against the real macOS-primary Node constraints the feasibility report surfaced (sandbox-exec deprecated; App Sandbox is GUI-only; Node 24's permission model does not propagate to spawned children), then record the selection, the threat model it actually delivers using NFR-001's exact scoping, and the Docker-dependency answer. The spike does not re-open the SPK-1 host-owns keystone (issue #598); it builds on it, because that keystone is precisely what makes a capability broker the natural enforcement layer.

Constraints grounding every finding below:

- **No supported macOS OS sandbox for a headless Node server process.** `sandbox-exec` (the `seatbelt` profile front-end) is deprecated and undocumented; macOS App Sandbox entitlements target App-Store-distributed GUI applications, not a long-lived headless Node process spawned by a local Electron app. (Feasibility report, Security dimension; sources cited there: macOS sandbox-exec deprecation reports.)
- **The Node 24 permission model is immature for this job and does not propagate.** Node's `--permission` model (the renamed, still flag-gated successor to the experimental permission model) can restrict the Node process's own `fs` / `child_process` / `worker` access, but it is process-local: it does **not** propagate to a spawned Docker container, a `docker compose` invocation, or a shell child, which run as ordinary OS processes outside Node's policy. A plugin that shells out or talks to the Docker daemon escapes Node's own permission model entirely. (nodejs.org Node permissions docs, cited in the feasibility report.) So the Node permission model can harden the host or a plugin's own Node process, but it cannot by itself be the enforcement boundary for the privileged operations component plugins actually perform (spawning processes, bringing up compose projects).
- **Overhead figures below are qualitative and reasoned, not benchmarked.** No micro-benchmark was run in this spike; the per-plugin overhead column is an engineering estimate from each mechanism's known startup and steady-state cost, and is labelled as such. A measured comparison, if needed, belongs to the F2.3 build, not this decision.

Maturity note carried verbatim into the decision: the **Node 24 permission model is not mature enough to be the enforcement boundary**, and critically it **does not propagate to spawned Docker/shell children**, so it cannot police the operations component plugins actually perform. It is usable as a _secondary hardening_ layer on the plugin's own Node process (restricting what the plugin's process can do with native `fs` / `child_process` _before_ it ever reaches the broker), which narrows the determined-attacker bypass surface, but it is not, and cannot be, the primary mechanism. The primary mechanism is the broker; the Node permission model and the container/VM tiers are additive.

## AC5: the selection is consistent with the SPK-1 host-owns / capability-broker direction

**Selected backend: the capability-broker model (option e) as the PRIMARY enforcement mechanism, with container-per-plugin (Docker where already present, or the Apple `container` framework / `Virtualization.framework` where available) as OPT-IN defense-in-depth, and Docker explicitly NOT required.**

This is consistent with SPK-1 (issue #598, the host-owns-vs-plugin-owns keystone) by construction, and deliberately does not re-open it:

- SPK-1's host-owns decision establishes that the host owns every process and container handle, and that both the declarative `LifecycleEngine` and the imperative hooks funnel every privileged operation through the single `HostComponentBroker` (architecture.md Decision summary; "The broker is the single privileged choke-point" under Security & compliance). A capability broker enforcing declared permissions is therefore not a new mechanism bolted on: it is the v2 `PermissionEnforcer` attaching policy at the choke-point the host-owns design already routes every privileged call through.
- Because there is exactly one privileged layer, enforcement (NFR-001), audit (FR-019), and cleanup (NFR-003) all attach at the same place. That is the structural reason the architecture states host-owns "was required and autonomous-runtime was rejected." Selecting the broker as the primary enforcement backend is the v2 realisation of that same keystone, not a divergence from it.
- The container/VM tiers sit _outside_ and _around_ the broker as additional OS boundaries, so adopting them never moves the enforcement choke-point or contradicts host-owns; they harden the same model rather than replacing it.

So the keystone is honoured, not re-litigated: the broker is the enforcement layer; the OS-isolation tiers are optional reinforcement where the host capability exists.

## AC2: the threat model the chosen backend actually delivers (NFR-001 scoping, verbatim)

Using NFR-001's exact scoping language, the selected backend delivers, and only claims, the following:

- **In scope (what it contains):** the threat model is scoped to **containing accidental damage, honest plugins, and casual abuse**. The broker denies any privileged operation outside a plugin's declared categories (`network` / `credentials` / `filesystem` / `processes` / `ports` / `docker`) and audit-logs it (FR-019). An honest-but-buggy plugin that asks for an undeclared action is blocked at the choke-point; a plugin that accidentally targets the wrong path/port/container is contained because the host, not the plugin, performs the privileged op; casual abuse (a plugin trying something it did not declare) is denied and recorded.
- **Out of scope (what it does not claim):** **resistance to a determined attacker** is explicitly NOT delivered by the broker alone. NFR-001 itself states that "resistance to a determined attacker requires the OS-isolation backend chosen by the sandboxing spike." Under the primary (broker-only) tier, a determined attacker authoring a malicious plugin can call native Node APIs (`fs`, `child_process`, raw sockets) directly inside the plugin process, bypassing the broker, exactly the residual gap the host-API-enforcement limitation in the feasibility report names ("a plugin owning its own children with native Node APIs bypasses it entirely"). Closing _that_ gap is what the **opt-in container/VM tier** is for: when the plugin runs inside a Docker / Apple `container` / `Virtualization.framework` boundary, native API calls are confined to that boundary, raising the bar toward (though, per NFR-001's scope, never absolutely guaranteeing) determined-attacker resistance.

In one line: **broker = accidental damage + honest plugins + casual abuse, guaranteed and unconditional; opt-in OS-isolation tier = the path toward determined-attacker resistance, available only where the host has the capability.** This is precisely the scoping NFR-001 and FR-018 already assert; the spike does not widen the claim.

## AC3: Docker dependency, stated explicitly

**Docker is NOT required.** The primary enforcement mechanism (the capability broker) needs no container engine, no VM, and no OS sandbox; it is an architectural property of the host-owns design and runs on every macOS-primary Node host unconditionally. This satisfies FR-018 ("not hardwired to Docker") directly and satisfies the NFR-005 host-capability gate (a host with no Docker still enforces declared permissions at the broker).

The Apple `container` framework and `Virtualization.framework` can each provide an OS-level isolation boundary **without any Docker dependency** where the host supports them, so the optional defense-in-depth tier is _also_ not Docker-bound. The host degrades across four rungs by host capability, highest-isolation first (the top three are the OS-isolation tier; the fourth is the broker-only floor):

1. **`Virtualization.framework` per-plugin VM** (where present and the deployment wants the strongest boundary) - heaviest, strongest, no Docker.
2. **Apple `container` framework** (macOS 15+, Apple silicon) - strong, no Docker.
3. **Docker container-per-plugin** (only where a Docker engine is already installed and running, which Roubo's database components often but not always imply) - good, _this_ rung is the Docker one.
4. **Broker-only** (the unconditional floor) - always available, no external dependency.

The host selects the highest rung its capability supports (queried, not assumed: NFR-005's `host.capability.query` gate), and falls back to the broker-only floor when none of the isolation runtimes are present. So: Docker satisfies FR-018/NFR-001 as _one optional rung_, but the Apple `container` framework and `Virtualization.framework` satisfy the OS-isolation goal on macOS _without_ Docker, and the broker satisfies the NFR-001 floor with no container runtime at all. The answer to the issue's explicit question is therefore: **no Docker dependency; the Apple container framework / Virtualization.framework (and, as the floor, the broker itself) satisfy NFR-001/FR-018 on macOS without one.**

## AC4: location and what F2.3 implements against

This document lives at `.specifications/component-plugins/spikes/spike-599-v2-plugin-isolation-backend.md`, the canonical spike location for the `component-plugins` feature folder, naming-consistent with `spike-553-...` under `.specifications/cut-list-improvements/spikes/` and `spike-407-...` under `.specifications/testbench/spikes/`.

The v2 F2.3 `PluginIsolationSandbox` slice (architecture.md Components table and Phase mapping, Phase v2) implements against this decision as follows, and pairs with the `PermissionEnforcer` it is specced alongside:

- **`PermissionEnforcer`** (architecture.md: "Intercepts every broker call, denies any outside the plugin's declared categories, and writes the audit entry") is the _primary_ mechanism this spike selects. It is the enforcement layer the `HostComponentBroker` exposes: a policy check inside the broker, before any call reaches `process-manager` / `docker` / `port-allocator`, that compares the requested op against the plugin's declared permission categories and denies + audit-logs (FR-019) anything outside them. F2.3's must-build floor is this broker-attached enforcer; it needs no OS sandbox to ship and is what makes NFR-001's v2 enforcement true on every host.
- **`PluginIsolationSandbox`** is the _optional, capability-gated_ OS-isolation tier this spike scopes: the host runs a plugin inside the highest-isolation runtime it supports (`Virtualization.framework` VM, Apple `container`, or Docker container-per-plugin), and degrades to broker-only enforcement where none is present. The architecture's "backend chosen by the sandboxing spike ... not Docker-locked" is resolved here: the backend is _capability-broker-primary with an opt-in container/VM tier_, the tier is selected at runtime by host capability (not hardwired), and Docker is one optional rung, never the requirement.

The `PermissionEnforcer` + `PluginIsolationSandbox` split per architecture.md is exactly the primary/defense-in-depth split this spike selects: the enforcer is the always-on broker-layer guard, the sandbox is the where-available OS boundary that hardens against the determined-attacker case the enforcer alone cannot.

## Recommendation

**Adopt.** Select the **capability-broker model as the primary v2 enforcement backend** (realised as the broker-attached `PermissionEnforcer`), with **container-per-plugin as opt-in, capability-gated defense-in-depth** (`Virtualization.framework` / Apple `container` / Docker, highest-isolation-first, broker-only floor), and **Docker explicitly not required**. Carry these into the v2 F2.3 build:

1. Build the `PermissionEnforcer` as the always-on floor: a policy check inside `HostComponentBroker` that denies any privileged op outside the plugin's declared categories and audit-logs it (FR-019). This ships independent of any OS sandbox and makes NFR-001's v2 enforcement true on every macOS-primary Node host.
2. Build the `PluginIsolationSandbox` as the capability-gated OS tier: select the highest-isolation runtime the host supports at runtime via the NFR-005 `host.capability.query` gate, and degrade to broker-only when none is present. Do not hardwire Docker; treat it as one optional rung.
3. Use the Node permission model only as _secondary_ hardening of the plugin's own Node process (it does not propagate to Docker/shell children, so it cannot be the boundary), never as the primary mechanism.
4. State the threat model to consumers exactly as NFR-001 scopes it: accidental damage / honest plugins / casual abuse are contained unconditionally; determined-attacker resistance is available only on the opt-in OS-isolation tier and is never claimed for the broker-only floor.

This gives F2.3 a fixed backend decision to build against, keeps it consistent with the SPK-1 host-owns keystone (issue #598), and honours FR-018's not-Docker-hardwired requirement and NFR-005's host-capability gate.

## Open questions / follow-ups

These are seams the upstream specs already track; this spike points the v2 work at them rather than resolving them here:

- **Trust tiers (bundled first-party vs locally-installed third-party plugins).** prd.md Open questions ("should bundled first-party component plugins and locally-installed third-party ones differ in the ... v2 enforcement default?") and feasibility Open questions both track this. It is **related but out of scope** for this spike: it bears on _when_ the opt-in OS-isolation tier is enabled by default (e.g. third-party plugins might default to the container/VM tier where available while first-party default to broker-only), but it does not change _which_ backend this spike selects. Resolve it as part of the v2 enforcement-default design, not here.
- **Whether the v2 sandbox needs a measured overhead benchmark before committing a default tier.** The overhead figures in AC1 are reasoned, not benchmarked (stated explicitly at the top). If the v2 build wants to default-enable an OS tier for any plugin class, a micro-benchmark of VM/container start and steady-state cost against the NFR-002 +500 ms p95 budget belongs to that build. This spike's decision (broker-primary, OS tier opt-in) does not depend on that measurement, because the floor carries no isolation-attributable overhead.
- **The exact `host.capability.query` shape for isolation-runtime detection.** Broker granularity and host-capability versioning are owned by the host-RPC spike (issue #600); this spike assumes that gate exists and uses it to select the isolation rung, but does not define its wire shape.

(No deferred work is named here without a tracked home: the trust-tiers item is an existing prd.md / feasibility open question owned by the v2 enforcement-default design; the benchmark and capability-query items are owned by the F2.3 build and the host-RPC spike (#600) respectively. Should the v2 build surface genuinely new deferred work, file a GitHub issue and reference the number inline per the repo's follow-up-references rule before relying on it.)
