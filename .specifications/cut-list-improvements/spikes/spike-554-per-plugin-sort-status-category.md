# Spike 554: Per bundled plugin, which sort fields can it apply source-side, and what is the only-to-do status-category mapping?

**Status:** Resolved · **Issue:** #554 · **Class:** decision · **Resolves:** prd.md FR-009/FR-010/FR-012/FR-014 (per-plugin sort + only-to-do mapping); architecture.md:151 Open question "Spike C" · **Implements:** CLI-FR-009, CLI-FR-014 · **Verified by:** CLI-TC-034, CLI-TC-043, CLI-TC-048, CLI-TC-052 · **Recommendation:** adopt

## Objective and method

Ordering (CLI-FR-009/CLI-FR-010) and only-to-do (CLI-FR-012/CLI-FR-014) both depend on per-plugin capability divergence. Jira has a rich JQL `ORDER BY` vocabulary and native status categories; the GitHub family (github-com and GHE) exposes only created/updated/comments sorts and an open/closed state model with no native "In Progress" category. This spike fixes those mappings so the planned `getSortFields` RPC and the only-to-do status defaults are honest per plugin, before either is implemented.

It is a **decision** spike: the deliverable is this findings doc, not code. Implementing the `getSortFields` RPC and the status defaults are their own downstream slices (Phase 3) and are explicitly out of scope here, as are any changes to `plugin-sdk/src/types.ts`, `ListIssuesParams`, or any plugin source.

Method: read the three bundled plugins' actual list paths and status-exclusion code as the source of truth, and derive each plugin's honest source-side capability from the request shape it already issues (not from the architecture's summary claim). The grounding files read:

- `plugins/jira-self-hosted/src/jql.ts` (the JQL builder, including the hardcoded `ORDER BY` tail and the category-first / name-fallback exclusion clause).
- `plugins/jira-self-hosted/src/__tests__/status-exclusion.test.ts` (the already-exercised `statusCategory` 400 -> name-list fallback behaviour).
- `plugins/github-com/src/methods/list-issues.ts` and `plugins/github-com/src/github-fetchers.ts` (the REST `/issues` + `/search/issues` request params, and the Projects v2 GraphQL shape).
- `plugins/ghe/src/methods/list-issues.ts` (a near-identical port of github-com's list path).
- `plugins/_shared-github/src/status-exclusion.ts` (the github-family name-only status exclusion).

A capability is only listed below if a source-side mechanism for it already exists in the read code or in the documented API surface the code calls; nothing is asserted purely from the architecture memo.

## Findings per acceptance criterion

### AC1: Sortable fields each plugin can honour source-side

Each row is tied to the concrete source-side mechanism that would carry it. "Source-side" means the ordering is applied by the external system as part of the query, so it is stable across pages (CLI-FR-010), not re-sorted client-side (which the PRD excludes for cross-page correctness).

| -------------------- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **jira-self-hosted** | `key`                              | JQL `ORDER BY key ASC\|DESC`                                                                                 | The PRD default sort ("by item key", CLI-FR-010). JQL sorts by issue key natively; no other plugin can.                                                                                                                                                                                                                                                                                   |

The RPC generalises this single fixed clause into a chosen field/direction.                                                                                                                                                                     |

**GitHub Projects v2 (both github-com and ghe) caveat.** When a source is a Projects v2 board (the `listFromProject` path), the plugin fetches the board's items via GraphQL and paginates **in-memory** over `visibleNodes` (list-issues.ts: `slice(offset * pageSize, ...)`). The board nodes carry `createdAt`, `updatedAt`, and `comments.totalCount` (github-fetchers.ts GraphQL selection), so created/updated/comments remain expressible, but the ordering would be applied client-side over the already-fetched node set, not pushed to the source. That is acceptable here only because the whole board is materialised before slicing; it is not the same "source-side, stable-across-pages" guarantee the REST path gives. The honest enumeration above is for the REST `/issues` path; the Projects v2 path can match the same field set but via post-fetch ordering of the full node list.

**GHE parity is gated on the FR-016 facet-contract prerequisite.** GHE's sort set matches github-com's only once GHE is brought up to the host-API 1.1.0 filter-facet contract (`filterFacets` / `getFacetOptions`), which it currently lacks. CLI-FR-016 is the confirmed prerequisite (Spike B, resolved): until GHE implements that contract, its `getSortFields` and source-side facet exclusion cannot reach parity. So the GHE row above is "the target set after FR-016 lands", not a capability available on GHE today.

**Conclusion (AC1).** Jira: `key`, `updated`, `created`, `priority`, `rank` (rank instance-conditional). github-com and ghe: `created`, `updated`, `comments`, with **no native key sort**. The host renders the picker from whatever set the plugin declares; for the github family that set never contains `key`, so the host's key-ascending default cannot apply there and the github family falls back to its natural order per CLI-FR-011.

### AC2: Only-to-do status mapping, decided per plugin

The only-to-do default (CLI-FR-012) excludes both the In Progress and Done status categories in-query, showing only items that are genuinely ready to pick up. How "In Progress + Done" maps differs by plugin because their status models differ.

**jira-self-hosted: exclude the In Progress and Done categories natively.** Jira has three native status categories ("To Do", "In Progress", "Done"). The builder already emits a category-first exclusion clause: on a supporting instance `buildExclusionClause` produces `statusCategory not in (...)` from `excludedStatusCategories` (jql.ts:108-112), ANDed across the whole source union so an excluded issue never occupies a result page. The only-to-do default is therefore expressed faithfully and natively by setting `excludedStatusCategories` to **both** `In Progress` and `Done` (today the shipped default excludes only `Done`; CLI-FR-012 is the deliberate flip, mitigated by the CLI-FR-018 migration notice). No approximation is needed: Jira has a real "In Progress" category to exclude.

**github-com and ghe: stay Closed-only by default; do NOT synthesise an implicit In Progress.** The github family has no `statusCategory` concept. Its only intrinsic state axis is open/closed, and the list paths already drop closed/non-open issues source-side (`normalizeState(...) !== "open"` in `listFromProject`, and `state: "open"` / `is:open` in `fetchIssuesPage`). So "exclude Done" is already satisfied: closed issues never appear. The open question is whether "exclude In Progress" can be honoured. The only status signal richer than open/closed is a **GitHub Projects v2 "Status" column** value (e.g. "In review", "PR open", "Done"), which `_shared-github/src/status-exclusion.ts` (`isStatusExcluded`) matches against the host-resolved `excludedStatuses` name list, case-insensitively and trimmed (status-exclusion.ts:16-24).

The decision, per CLI-FR-014, is that the github family stays **Closed-only by default** and does **not** apply a lossy implicit In Progress mapping. The reasoning (FR-014: "the default stays Done-only when no faithful mapping exists"):

- The Projects v2 "Status" column is **user-defined and free-form**. Its column names are arbitrary per board ("In review", "PR open", "Blocked", "Doing", a team's own labels); there is no canonical, API-guaranteed "In Progress" category the plugin could exclude by definition. Treating some heuristic subset of column names as "In Progress" would be a guess that silently hides actionable to-do items on boards that happen to name a column that way, or miss in-progress work on boards that name it differently.
- The "Status" column exists **only on the Projects v2 path**, not on the plain repo `/issues` path. A repo source has no status column at all, so there is nothing to map; the In Progress concept simply does not exist there.
- Excluding by an arbitrary status name is therefore opt-in configuration (`excludedStatuses`), not a default. The honest default is: github family excludes closed/done (it always did, via the open-only fetch) and leaves In Progress alone, because there is no faithful, board-independent native category to exclude. This is exactly the CLI-FR-014 carve-out ("maps the to-do exclusion to its closest available approximation, and the default stays Done-only when no faithful mapping exists").

So the per-plugin only-to-do default is:

| Plugin           | Default exclusion                                                       | Mechanism                                                                                           | Faithful In Progress mapping?                            |
| ---------------- | ----------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- | -------------------------------------------------------- |
| jira-self-hosted | In Progress **and** Done categories                                     | `statusCategory not in ("In Progress", "Done")` (jql.ts)                                            | Yes (native category)                                    |
| github-com       | Closed/Done only (open-only fetch); In Progress not excluded by default | `state: "open"` / `is:open` already; optional `excludedStatuses` name match on Projects v2 "Status" | No (Status column is free-form, repo path has no status) |
| ghe              | Same as github-com                                                      | Same as github-com                                                                                  | No                                                       |

### AC3: Jira instance WITHOUT statusCategory JQL support, the status-name fallback

Some Jira instances (older / certain configurations) reject `statusCategory` in JQL with a 400 ("Field 'statusCategory' does not exist or you do not have permission to view it."). The builder already handles this: `BuildJqlInput` carries `statusCategorySupported` (default `true`) and a separate `excludedStatuses` **name** list. When `statusCategorySupported` is `false`, `buildExclusionClause` emits `status not in (...)` from the resolved name list instead of the category clause (jql.ts:103-116), and the category list is ignored on that path.

The behaviour, exercised verbatim in `plugins/jira-self-hosted/src/__tests__/status-exclusion.test.ts`:

1. **First attempt uses the category form.** A supporting instance: one JQL with `statusCategory not in ("Done")`, no `status not in` (test "uses statusCategory exclusion on a supporting instance"). For the only-to-do default this category list becomes `("In Progress", "Done")`.
2. **On a 400, retry with the status-name list.** The plugin flips `statusCategorySupported` to false for that instance and rebuilds: the retry emits `status not in ("Closed", "Done", "Resolved")` and contains no `statusCategory` (test "retries with status-name exclusion when statusCategory is rejected (TC-037)"). So the only-to-do default on an unsupported instance is expressed as an explicit list of **status names** that stand in for the In Progress + Done categories.
3. **The unsupported verdict is cached per instance.** After the first 400 -> name retry, later polls build the name form directly and never pay the 400 round-trip again (test "caches the unsupported instance so later polls skip the 400 round-trip"). The cached flag is per instance, so one unsupported instance does not taint others.
4. **The fallback decision logs only a safe enum.** It logs `status-names-fallback` with no JQL fragments, no PAT, no project key, no issue content (test "logs the fallback with no JQL, search term, PAT, or issue content", NFR-003).

**The implication for only-to-do on an unsupported instance (the open spec question, prd.md:120).** Because category JQL is unavailable, "exclude In Progress + Done" can only be honoured by an explicit list of the instance's actual in-progress and done **status names** (e.g. "In Progress", "In Review", "Closed", "Done", "Resolved"). A category-to-names default list is a best-effort seed; a faithful mapping for a non-standard workflow ultimately needs the user to configure the exact in-progress status names for that instance. This spike does not change that mechanism; it specifies that the only-to-do default rides the existing `excludedStatuses` fallback list on unsupported instances and that, where a workflow uses non-standard status names, the name list is the configuration surface (it is not silently inferred).

## Recommendation

**Adopt.** The per-plugin capability split is settled and ready to carry into the Phase 3 implementation slices without further research:

1. **`getSortFields` declared sets (CLI-FR-009).** jira-self-hosted declares `key`, `updated`, `created`, `priority`, `rank` (gate `rank` on instance support, see Open questions). github-com and ghe declare `created`, `updated`, `comments` and **no `key`**. github-com/ghe apply the chosen field via the REST `sort`/`direction` (and `order`) params they already pass; jira-self-hosted generalises the hardcoded `ORDER BY updated ASC` tail to a chosen field/direction.
2. **Only-to-do defaults (CLI-FR-012 / CLI-FR-014).** jira-self-hosted defaults to `excludedStatusCategories = ["In Progress", "Done"]` (the deliberate flip from today's Done-only). github-com and ghe stay Closed-only by default (their open-only fetch already excludes Done) and apply **no** implicit In Progress mapping; In Progress exclusion remains opt-in via the `excludedStatuses` name list against the Projects v2 "Status" column, because that column is free-form and absent on the repo path.
3. **Unsupported-Jira fallback (CLI-TC-037).** Reuse the existing `statusCategory` 400 -> `excludedStatuses` name-list fallback as-is; the only-to-do default is expressed as a best-effort name list on those instances, with exact in-progress status names being user-configurable where the workflow is non-standard.

No host-API or plugin source changes are made by this spike; it fixes the contract the downstream slices implement.

## Open questions and follow-ups

- **Jira `rank` availability is instance-conditional.** `ORDER BY Rank` requires Jira Software (the rank custom field) to be present. `getSortFields` should either probe for the rank field or degrade gracefully (omit `rank` when absent) rather than declare it unconditionally. This is an implementation detail for the Phase 3 Jira `getSortFields` slice, not a decision this spike needs to settle. If it is deferred into that slice, it should be tracked on that slice's issue rather than left as a bare "follow-up".
- **github-family default sort direction.** github-com/ghe have no `key` sort, so the host's "ascending by item key" default (CLI-FR-010) cannot apply. The natural fallback (per CLI-FR-011) is the plugin's own order; the github REST default the code already uses is `updated`/`desc`. Whether the host should present `updated` (or `created`) as the github-family default sort, or render no picker and use natural order, is a host-side UX decision for the sort-picker slice (CLI-FR-009/CLI-FR-011), not a plugin capability question.
- **Projects v2 ordering is post-fetch, not source-side.** On the Projects v2 path the plugin materialises the whole board then slices, so any sort there is applied over the fetched node set, not pushed to the source. This satisfies cross-page stability only because the full board is loaded before paging; if board sizes ever exceed what is materialised in one pass, this assumption needs revisiting.
- **Only-to-do on unsupported-Jira non-standard workflows.** The name-list fallback is best-effort; a workflow with non-standard in-progress status names needs the user to configure the exact names. Whether the host should surface a "configure your in-progress status names" affordance for such instances (vs silently under-excluding) is a config-UX follow-up for the status-exclusion slice (CLI-FR-013/CLI-FR-017). (Open spec question, prd.md:120.)
