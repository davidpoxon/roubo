# Spike 553: What is the authoritative cache-key composition, invalidation matrix, and lifecycle-eviction contract for the cut-list persistent warm cache?

**Status:** Resolved · **Issue:** #553 · **Class:** decision · **Resolves:** feasibility Spike A (cache-invalidation correctness, the DE-RISK item) and architecture Open question "Spike A" · **Implements:** CLI-FR-003, CLI-FR-004, CLI-NFR-001 · **Verified by:** CLI-TC-003, CLI-TC-005, CLI-TC-008 · **Gates:** Phase 2 cache slice (CutListQueryService + DiskSnapshotStore) · **Recommendation:** adopt

## Objective and method

Pin down, before any cache code is written, the three things the Phase 2 cache slice (CutListQueryService + DiskSnapshotStore, architecture.md components table) must build against:

1. the exact, frozen cache-key field set and its canonicalisation rule (CLI-FR-003),
2. the invalidation test matrix the cache slice's `makeKey` and load path are tested against (CLI-FR-003, CLI-FR-004), and
3. the lifecycle-eviction contract: which events evict, by which mechanism, and the disable-vs-keep-warm decision, plus the eviction policy thresholds and the `0600` / no-secrets security rules (CLI-FR-004, CLI-NFR-001).

This is a decide-and-document spike. No `server/` code is written here; the cache slice itself is explicitly out of scope (issue #553 Out of Scope). The deliverable is this findings doc.

Method: take the field set and shapes already drafted in `architecture.md` (CacheKey at the data-model table, DiskCacheEntry alongside it, the eviction hooks under Interfaces, the security invariants under Security and compliance), reconcile them against the PRD requirements (CLI-FR-003, CLI-FR-004, CLI-NFR-001) and the open questions the PRD carries (eviction-policy specifics, disable-vs-keep-warm), then fix each decision verbatim and trace every invalidation row to the authored test cases (CLI-TC-003, CLI-TC-005, CLI-TC-008). Decisions are grounded in the real repo primitives the cache will reuse: `atomicWrite(filePath, data, mode?)` in `server/services/state.ts:53` (already takes a `mode` argument, so `0600` is a call-site concern, not a new primitive), the `~/.roubo/` directory convention (`state.ts:25-51`, `ensureDirs`), Node's built-in `crypto` for hashing (no new dependency, consistent with the architecture's zero-new-dependency stance), and the existing in-memory `issue-snapshot-cache.ts` whose composite-key and clear-on-uninstall behaviour the disk store extends.

## Findings per acceptance criterion

### AC1: the cache-key field set is fixed

**Verdict: fixed. Twelve fields, listed below verbatim, canonicalised to a stable string then SHA-256-hashed to produce the on-disk filename.**

The CacheKey is the structured object; the on-disk artifact is named by its hash. The object is serialised by emitting every field below in the fixed declaration order given here (never relying on JavaScript object key-insertion order), with each field canonicalised per its rule, and the resulting UTF-8 string is hashed with `crypto.createHash("sha256")`. The hex digest (optionally truncated to a fixed prefix that still comfortably clears collision risk for a single local user, e.g. the full 64 hex chars) is the filename stem under the per-project cache directory. Hashing is required, not optional: a raw composite key would exceed filesystem name-length limits once `sources` and filters are included (PRD open question, line 119), and a hash sidesteps path-injection concerns on user-controlled strings (search text, source ids).

The field set, in canonical serialisation order:

This is exactly the field set CLI-FR-003 enumerates and the CacheKey row in `architecture.md` drafts; this spike freezes it and adds the canonicalisation rules and ordering, which the architecture left implicit.

**Excluded from the key, deliberately:** `cursor` (the snapshot is first-page-only, so the key is always the page-1 query; a cursor never enters it), and anything credential-bearing (tokens stay in the OS keyring via `credential-store.ts`; the endpoint is hashed, not embedded). Response data and `capturedAt` are properties of the entry, not the key.

**`cacheSchemaVersion` bump strategy.** `cacheSchemaVersion` starts at `1`. Increment it (by exactly one) whenever a change would make a previously written file mis-parse or carry stale-shaped data under the new code, specifically: (a) the DiskCacheEntry envelope shape changes (a field added, removed, renamed, or re-typed), (b) the canonicalisation rule for any keyed field changes (e.g. a new sort applied to `sources`, a different `null`-collapse rule) such that the same logical query would now hash differently, or (c) the cached `PaginatedIssues` payload shape changes in a way the reader cannot tolerate. Do not bump for additive, backward-tolerant changes the reader already ignores. On load, the store compares the file's stored `cacheSchemaVersion` (and `pluginVersion`) against the current values; a mismatch is treated as a cold miss and the file is discarded (overwritten on the next successful `put`), never an error. The version lives both in the key (field 12, so a bump changes the filename and the old file simply stops being addressed) and inside the DiskCacheEntry envelope (so a stale file that happens to collide on name is still rejected on read). Belt and braces: the key bump means new and old entries never share a filename; the envelope check means even a hand-tampered or hash-colliding file is rejected.

### AC2: the invalidation test matrix is written

**Verdict: written. One row per key input asserting change leads to miss, plus the two load-time rejection rows (older schema_version, corrupt/partial file). Each row names the assertion the cache slice's tests make and traces to an authored test case.**

The matrix is the contract the Phase 2 `makeKey` and `DiskSnapshotStore.get` are tested against. "Miss" means: the new key hashes to a different filename, so `get` returns `null` and the prior entry is neither read nor overwritten (it remains addressable under its own key until evicted or aged out). Setup for every key-input row: a warm entry exists for a baseline key; change exactly one input; assert a miss and that the baseline entry is untouched.

The hit-side counterpart (verified once, so the suite distinguishes a real miss from an over-eager one): re-issuing the **same** logical query (including `sources` and `labels` in a different order) produces the same hash and a hit. This guards against canonicalisation that is too coarse (would never hit) or too fine (would leak across tenants).

### AC3: the eviction trigger set is enumerated, with the disable-vs-keep-warm decision recorded

**Verdict: five triggers enumerated, each mapped to its mechanism. Decision: plugin disable EVICTS the disk cache (does not keep it warm). Rationale below.**

Two mechanisms exist. (1) **Explicit hook**: a lifecycle event calls `DiskSnapshotStore.evictPlugin(pluginId)` or `evictProject(projectId)`, which deletes the affected files (per-project subdirectories make `evictProject` a single directory removal). (2) **Self-invalidation**: the event changes a CacheKey field, so the next read hashes to a new filename and the old entry is simply never addressed again (it then ages out under the AC4 policy). Self-invalidation alone does **not** delete the old file, so for any event that must remove private content from disk promptly (a security-driven clear), an explicit hook is required even when the key would also change.

| #   | Trigger                                                           | Mechanism                                                                      | Notes                                                                                                                                                                                                                                                                                                                                                 |
| --- | ----------------------------------------------------------------- | ------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| E1  | Plugin uninstall                                                  | explicit `evictPlugin(pluginId)`                                               | Mirrors the existing `issue-snapshot-cache.clearSnapshot(pluginId)` call already wired into `plugin-manager.ts:1034` on uninstall. A re-installed id is a different deployment and must not inherit prior cached issues.                                                                                                                              |
| E2  | Plugin disable                                                    | explicit `evictPlugin(pluginId)`                                               | **Decision point. The disk cache evicts on disable (see below).** This deliberately diverges from the in-memory cache, which keeps its snapshot on disable on purpose.                                                                                                                                                                                |
| E3  | Project unregister                                                | explicit `evictProject(projectId)`                                             | Wired from `project-registry.unregisterProject` per the architecture's eviction-hooks contract. Single directory removal.                                                                                                                                                                                                                             |
| E4  | Plugin version change                                             | explicit `evictPlugin(pluginId)` AND self-invalidation                         | `pluginVersion` is key field 2, so a new version self-invalidates (new filename); the explicit hook additionally deletes the stale prior-version files so old private content does not linger on disk. | E5  | Integration reconfiguration (endpoint/instance or sources change) | self-invalidation (key changes) PLUS an explicit evict of the superseded entry | `instanceHash`/`sources` are key fields 3/5, so the new config misses correctly (CLI-TC-005 S001). But CLI-TC-005 S002 also asserts the OLD instance entry is evicted and the eviction is observable in logs, attributed to integration reconfiguration. So reconfiguration must explicitly evict the prior entry, not merely orphan it for the ager. |

**Disable-vs-keep-warm decision (resolves PRD open question, prd.md line 123): evict on disable.**

The in-memory `issue-snapshot-cache.ts` deliberately keeps its snapshot when a plugin is disabled (`plugin-manager.ts` comment at the uninstall site: "we deliberately do not clear the snapshot on disable(): FR-014 calls for serving the last-good snapshot while a plugin is disabled"). That keep-warm behaviour is correct **for the in-memory cache**, because its whole job is the errored/disabled stale-fallback serve path (CLI-NFR-006, the today-banner path), and an in-memory buffer is process-scoped, never on disk, never `0600`-relevant, and gone on restart.

The **disk** cache is a different object with a different invariant. It is, per `architecture.md` Security and compliance, "the first place private GHE/Jira issue content (titles, bodies, assignees) is written to local disk." CLI-NFR-001 names plugin disable explicitly in its clearing list ("cache entries are cleared on plugin uninstall, plugin disable, project unregister, plugin version change, and integration reconfiguration"). The security invariant is the tie-breaker: when a user disables a plugin, the expectation is that its data stops being live; leaving private issue content in a `0600` file on disk for an indefinite re-enable window is exactly the exposure NFR-001 exists to prevent, and the file persists across restart (unlike the in-memory buffer), widening the window. The convenience of a warm re-enable does not outweigh the standing on-disk exposure for a local-tool threat model where the user can simply re-fetch on re-enable (cold load is bounded by the external system, not this cache, and the only cost is one round-trip).

Therefore: **disk cache evicts on disable (E2); the in-memory cache keeps warm on disable (unchanged).** The two caches coexist precisely because they have opposite disable semantics: the in-memory buffer continues to serve the disabled-plugin stale-fallback page (CLI-NFR-006), while the disk warm cache is cleared to honour NFR-001. The architecture's open question "single authority for the disk-store hit vs the in-memory errored-fallback ... and when to retire the in-memory cache" is the seam where this coexistence is eventually reconciled; until the disk store demonstrably covers the errored-fallback serve, the in-memory cache stays and the disable-keep-warm fallback rides on it, not on disk.

This is a divergence the cache slice must implement deliberately: do not copy the in-memory cache's "skip clear on disable" line into the disk store. The disk store's disable path calls `evictPlugin`.

### AC4: the eviction policy and the security rules are specified

**Verdict: concrete thresholds chosen below, anchored to the PRD's ~50-200 KB/entry note and a single-local-user scale. The `0600` file-mode and no-secrets-in-payload rules are specified and tied to the real primitives.**

**Eviction policy (the size/age bounds the architecture and PRD left open, prd.md line 118).** The scaling unit is the number of distinct (project, plugin, sources, filters, sort, pageSize) combinations on disk; for one local user this is small (tens, not thousands). The PRD notes a 50-issue page with `raw` fields can be ~50-200 KB/entry. Chosen defaults, with rationale:

- **Per-entry size cap: 1 MB.** Comfortably above the 200 KB worst-case the PRD cites (roughly 5x headroom for an unusually heavy page or a future field addition), low enough that a single pathological entry cannot blow the directory budget. On `put`, if the serialised entry exceeds the cap, skip the write (do not persist) and log a skip event (CLI-NFR-009); a too-large snapshot is simply not cached, which degrades to a cold load, never an error.
- **Total directory size bound: 50 MB.** At ~200 KB/entry this is roughly 250 worst-case entries, far more than the realistic distinct-query count for one user, so the bound is a safety ceiling against unbounded growth (e.g. churny sort/filter exploration), not a routine constraint. When a `put` would exceed the total bound, evict least-recently-used entries (by file mtime, refreshed on read-hit) until the write fits.
- **Max age: 7 days.** A warm hint older than a week is almost certainly going to be revalidated on next open anyway (stale-while-revalidate serves it once then refreshes), and a 7-day ceiling bounds how long any private snapshot can sit on disk without the user touching that query. Entries past max age are treated as a cold miss on read (discarded, not served) and swept opportunistically on `put`. Age is measured from `capturedAt`.

Enforcement is opportunistic on `put` (no background timer, no new process machinery, consistent with the hand-rolled-store, zero-new-dependency stance): each write first sweeps aged-out entries, then enforces the total bound by LRU eviction, then writes (or skips, if over the per-entry cap). Reads additionally treat an over-age entry as a miss so a stale file is never served even before the next `put` sweeps it. These three thresholds (1 MB / 50 MB / 7 days) are the defaults the cache slice ships; they are config-free for v1 (no user-facing knob), matching the local-tool, single-user framing.

**Security rules (CLI-NFR-001), tied to the real primitives:**

- **File mode `0600`.** Every entry is written through a single private helper in DiskSnapshotStore that calls `atomicWrite(filePath, data, 0o600)`. `atomicWrite` (`server/services/state.ts:53`) already accepts a `mode` argument and applies it to the temp file before the atomic rename, so `0600` is purely a call-site obligation: pass `0o600`, never omit it (the default is `0o666`). A dedicated unit test asserts the on-disk file mode is `0600` (architecture Security and compliance already commits to this test). The cache directory itself is created under `~/.roubo/` via the established `ensureDirs`/`mkdirSync` pattern (`state.ts:45-51`); create the cache subdirectory the same way.
- **No secrets in the payload.** The DiskCacheEntry payload is `{ cacheSchemaVersion, cacheKeyHash, capturedAt, pluginVersion, response: PaginatedIssues }` (architecture data-model). It carries issue content only; it must contain **no** credentials or tokens. Those remain in the OS keyring via `credential-store.ts` and never enter the cache write path. The CacheKey likewise embeds no secret: the endpoint is hashed into `instanceHash` (field 3), not stored raw. A unit test asserts a serialised entry contains none of the credential field names (architecture commits to this; CLI-TC-005 S002-O02 asserts no credential/token material appears in the cache file or logs).
- **Observability without leakage (CLI-NFR-009).** Eviction, corrupt-file discard, schema/version-mismatch discard, size-cap skip, and age sweep all emit a log event with enough context to diagnose (which trigger, which plugin/project) and **no** credential or issue-body material. Per repo testing rules, these intentional logs are spied and asserted in tests, not emitted to test stdout.

## Recommendation

**Adopt.** The cache-key field set (AC1), invalidation matrix (AC2), eviction-trigger set with the evict-on-disable decision (AC3), and the eviction-policy thresholds plus security rules (AC4) are fixed and internally consistent, and they trace cleanly to CLI-FR-003 / CLI-FR-004 / CLI-NFR-001 and to CLI-TC-003 / CLI-TC-005 / CLI-TC-008. Phase 2 (CutListQueryService + DiskSnapshotStore) has a fixed spec to build `makeKey` and `DiskSnapshotStore.get/put/evictPlugin/evictProject` against. Carry these forward into the cache slice:

1. Serialise the twelve fields in the declared order with the per-field canonicalisation rules (sort `sources`, sort and de-dup `labels` / `excludedStatus*`, collapse absent objects to `null`), then `crypto.createHash("sha256")`. Do not rely on object key-insertion order.
2. Store `cacheSchemaVersion` and `pluginVersion` both in the key and inside the entry; reject mismatches on read as a cold miss, never an error.
3. Implement the matrix's L1/L2 rows as the load path's first defence: every read is `try/catch` returning `null`, so a corrupt or stale file is a cold miss that never reaches a fatal/startup error.
4. Wire `evictPlugin` on uninstall, disable, and version-change, and `evictProject` on project unregister; explicitly evict the superseded entry on integration reconfiguration (not only self-invalidation), so CLI-TC-005 S002's log-attributed eviction holds.
5. Implement disable as **evict** in the disk store (the opposite of the in-memory cache); do not copy the in-memory "skip clear on disable" behaviour.
6. Ship the 1 MB / 50 MB / 7-day thresholds as config-free defaults, enforced opportunistically on `put` with read-time age rejection; write every entry via a single helper hard-coding `atomicWrite(..., 0o600)`.

## Open questions / follow-ups

These are scoping seams the architecture already tracks as Open questions; this spike does not resolve them and points the Phase 2 work at them rather than inventing prose placeholders:

- **Single authority for the disk-store hit vs the in-memory errored-fallback** (architecture.md Open questions, the bullet beginning "Single authority for the disk-store hit"): the two caches coexist with opposite disable semantics (this spike, AC3); the architecture flags reconciling them and eventually retiring the in-memory cache as an open question for the Phase 2 build, not for this spike. The disk store's evict-on-disable decision here is the input to that reconciliation.
- **Stale-while-revalidate delivery mechanism** (architecture.md Open questions, "Stale-while-revalidate delivery"): client-refetch vs an SSE freshness push. This spike covers the cache key/invalidation/eviction contract only; the serving path is explicitly out of scope (issue #553 Out of Scope) and the architecture defaults to client refetch unless this spike says otherwise. This spike does not change that default.
- **Jira `statusCategory` JQL support fallback** and **GitHub Projects v2 In-Progress category** (architecture.md Open questions; PRD lines 120-121): these bound how `excludedStatusCategories` / `excludedStatuses` (key fields 7/8) are populated, but are Spike C territory (per-plugin sort + status mapping), not this spike. The cache key treats whatever exclusion lists the host resolves as opaque inputs, so Spike C's resolution does not change AC1's field set.
